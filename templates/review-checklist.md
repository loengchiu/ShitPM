# Review Checklist

## 评审基础信息

- 本轮评审对象是什么
- 实际采用了哪些文件路径
- 当前路径与 `stable_baselines` 是什么关系
- 本轮评审是自检模式还是多角色评审模式
- 若为多角色评审：默认角色组合、评审团类型、追加角色、省略角色及原因是否已说明
- 本轮结论来源是否明确（真实状态判断 / 上下文推断）

## Mindmap

- 是否覆盖范围、主流程、模块、系统结构、关键判断点、状态分支和待澄清问题
- 是否仍能代表当前业务范围和主流程
- 是否存在足以影响 `Feature List` 的结构性歧义
- 若需回写，是否明确回到 `mind`

## Feature List

- 是否使用页面/能力级颗粒度
- 是否避免步骤级描述
- 是否在模块级描述涉及角色
- 是否只保留能力边界，而未混入页面结构
- 版本边界是否清楚
- 关键需求摘要是否克制

## Page Structure

- 是否以页面目标和结构树为主体
- 是否只写页面与区域事实
- 是否避免过程性说明
- `对应功能点` 是否只做映射而不重复解释
- 页面目标是否清晰
- 每个区域是否都有作用、关键操作、关键状态

## PRD

- 是否基于 `Feature List + Page Structure`
- 是否具体可执行
- 是否存在空泛表述
- 是否保留了 `Feature List` 中的模块角色信息
- 是否能判断本期范围与版本边界
- 交互逻辑是否与业务复杂度匹配，而不是一律写得过重
- 简单交互是否只写到触发、反馈和结果即可
- 中高复杂交互是否已补足交互顺序、状态变化和必要分支
- 若模块内存在复杂表单，表单数据规范是否已直接写在该模块的详细需求说明里
- 是否需要回写上游

## Prototype

- 是否基于稳定 PRD 生成
- 是否继承统一视觉基线
- 每个关键页面是否能回溯到 `Page Structure` 的页面目标
- 每个关键区域是否能回溯到 `Page Structure` 的区域定义
- 是否出现了 `Page Structure` 中不存在的主页面或关键区域

## Prototype Annotation

- 是否已判定初始化标注或增量更新模式
- 同一模块是否只有一个角标，无重复
- 浮窗内容是否足以替代对应 PRD 段落阅读
- 是否满足：点击打开、拖拽、仅 X 关闭、页面切换后同步状态
- 若本轮涉及编号回写，是否已同步更新对应 PRD
- 是否存在需求冲突需回退到 `prd` 或 `mock`

## 回写判定

- 当前问题属于业务流程/结构、能力边界、页面承载还是 PRD 规格层
- 如需回写，是否已明确回写目标是 `mind`、`feat` 还是 `page`
- 如需回退，是否已明确停留当前阶段、回写上游或进入问答模式

## 版本与锚点关系

- 当前评审使用的 `Feature List / Page Structure / PRD` 是否为当前稳定锚点
- 是否存在新版已生成但 `stable_baselines` 仍指向旧版
- 如版本关系未判清，是否已停止继续给出跨产物结论

## 评审结论

- 最终结论是否明确为 `通过 / 带条件通过 / 阻塞`
- 若为带条件通过，是否列出条件项、责任人、最晚补齐时点
- 若为阻塞，是否列出阻塞问题、归属层级和推荐路由

---

## 多角色评审（角色专属视角）

### 评审团组合

- 当前采用的是哪类评审团：`需求边界评审团 / 页面承载评审团 / 可开发性评审团 / 上线风险评审团`
- 评审团是否与当前评审对象匹配
- 是否说明了本轮为何采用该组合

### 业务经理

- 业务目标是否被完整覆盖
- 优先级边界是否合理
- 本期范围的投入产出是否清晰

### 项目经理

- 是否存在跨团队接口依赖未明确
- 关键里程碑是否有对应的产物节点
- 是否有未解决的风险项

### 产品经理

- 需求是否完整，是否有遗漏场景
- 边界情况和例外路径是否已覆盖
- 版本范围是否已明确

### 前端

- 交互复杂度是否在合理范围内
- 是否存在难以实现的联动逻辑
- 组件复用性是否已考虑

### 后端

- API 接口面是否清晰
- 数据模型是否已明确
- 权限设计是否完整

### UI

- 视觉基线是否一致
- 设计稿与原型是否对齐
- 是否存在无法实现的视觉要求

### 测试

- 关键业务路径是否可测试
- 是否有明确的验收标准
- 边界情况是否已覆盖在测试范围内

---

## 问题定性与问答模式

- 是否存在需要用户逐条回答的问题
- 若问题较多，是否已先做归并、去重和阻塞度排序
- 是否只推进当前最关键的 1 个问题；如强相关，是否控制在同轮最多 3 个
- 若问题达到 `8+`，是否先给出问题地图与首要问题，而不是一次性全量追问
- 每条问题是否已先定性为 `真实风险 / 过度担忧 / 待观察项`
- `真实风险` 是否写入 `blockers` 或已当场修订
- `过度担忧` 是否仅记录说明而不阻塞
- `待观察项` 是否写入 `pending_confirmations`
- 若用户无法判定，是否按 `真实风险` 处理
- 当前问题回答后，是否重新判断其余问题是否仍需继续追问

## 与 fix 的联动

- 若评审问答已改变事实源，是否已明确从 `rev` 路由到 `fix`
- 是否已说明问题层级和建议回写目标
- 是否避免把正式变更处理停留在评审记录层
