# Review Checklist

## Feature List

- 是否使用页面/能力级颗粒度
- 是否避免步骤级描述
- 是否在模块级描述涉及角色
- 是否只保留能力边界，而未混入页面结构
- 版本边界是否清楚
- 关键需求摘要是否克制

## Page Structure

- 是否以页面目标和结构树为主体
- 是否只写页面与区域事实
- 是否避免过程性说明
- `对应功能点` 是否只做映射而不重复解释
- 页面目标是否清晰
- 每个区域是否都有作用、关键操作、关键状态

## PRD

- 是否基于 `Feature List + Page Structure`
- 是否具体可执行
- 是否存在空泛表述
- 是否保留了 `Feature List` 中的模块角色信息
- 是否能判断本期范围与版本边界
- 是否需要回写上游

## Prototype

- 是否基于稳定 PRD 生成
- 是否继承统一视觉基线
- 每个关键页面是否能回溯到 `Page Structure` 的页面目标
- 每个关键区域是否能回溯到 `Page Structure` 的区域定义
- 是否出现了 `Page Structure` 中不存在的主页面或关键区域

## Prototype Annotation

- 是否已判定初始化标注或增量更新模式
- 同一模块是否只有一个角标，无重复
- 浮窗内容是否足以替代对应 PRD 段落阅读
- 是否满足：点击打开、拖拽、仅 X 关闭、页面切换后同步状态
- 若本轮涉及编号回写，是否已同步更新对应 PRD
- 是否存在需求冲突需回退到 `pm-prd` 或 `pm-pt`

## 回写判定

- 当前问题属于能力边界、页面承载还是 PRD 规格层
- 如需回写，是否已明确回写目标是 `pm-fl` 还是 `pm-ps`

## 多角色评审（角色专属视角）

### 业务经理

- 业务目标是否被完整覆盖
- 优先级边界是否合理
- 本期范围的投入产出是否清晰

### 项目经理

- 是否存在跨团队接口依赖未明确
- 关键里程碑是否有对应的产物节点
- 是否有未解决的风险项

### 产品经理

- 需求是否完整，是否有遗漏场景
- 边界情况和例外路径是否已覆盖
- 版本范围是否已明确

### 前端

- 交互复杂度是否在合理范围内
- 是否存在难以实现的联动逻辑
- 组件复用性是否已考虑

### 后端

- API 接口面是否清晰
- 数据模型是否已明确
- 权限设计是否完整

### UI

- 视觉基线是否一致
- 设计稿与原型是否对齐
- 是否存在无法实现的视觉要求

### 测试

- 关键业务路径是否可测试
- 是否有明确的验收标准
- 边界情况是否已覆盖在测试范围内
