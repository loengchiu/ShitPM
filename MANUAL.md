# ShitPM 使用说明书：全链路 AI 产品经理工作流

## 🧭 一、 核心运行机制

### 1. 状态机与多级锚点 (State Machine & Baseline)
你不需要靠脑子记当前进度。系统靠底层的 `docs/project-status.json` 进行自驱动调度：
*   **`current_stage`**: 当前在哪个生产环节。
*   **三重稳定锚点 (`stable_baselines`)**: 
    1. 前一个技能留下的 **轻量简介**（用于范围）。
    2. 基于前者导出的 **功能清单 Feature List**（能力锚点）。
    3. 基于前者推演的 **页面结构 Page Structure**（页面锚点）。
下个环节永远**只能认**上个环节锁定的“稳态锚点”作为事实源。这就像合同一样严苛，彻底杜绝了 AI 到了后期胡乱发明新功能。

### 2. 门禁与阻断器 (Stage Gates & Blockers)
典型的瀑布流加防御机制。
如果上一层锚点没写、存在激烈的分歧碰撞、或发现跨域的模块数量爆炸，工作流会亮起红灯（`blockers`），拉动手刹强行停止，让你进行人工确认，而不是产生一堆废料发给研发。

## 🛠 二、 十大核心技能 (Skills) 详析与调遣

这些技能就是你调派遣 AI 苦力的命令，分为四个职能小队：

### 【调度纠偏队：负责统帅与逆向同步】
*   **🔑 `pm-go`（唯一主入口 / 项目管家）**
    *   **何时用：** 新需求落地，或者不知道下一步干什么时，无脑敲这个命令。
    *   **机制：** 系统自动分析你的库、目录和卡点状态，判断需不需要拆分子项目？需不需要分块写 PRD？确定后将你推进到正确的下一环节。
*   **🔄 `pm-fix`（神级特性：逆向热更同步引擎）**
    *   **何时用：** 当你不爽 AI 写的页面逻辑，你自己去 Markdown 里删了两段功能。
    *   **机制：** 修改后敲入，它绝不无脑重写全文。它像个剥洋葱的医生，分析你的改动：是“表达文案优化”、还是“改了视图结构（穿透回写 PS）”、还是“增加了核心能力边界（穿透回写 FL）”。**静默解决修改导致的前后隔离断层**。

### 【探索定义队：化混沌为具象】
*   **🔭 `pm-scope`（范围猎犬）**
    *   **何时用：** 在真正开工前，先框定“这个业务池有多大”。
    *   **机制：** 若覆盖多个独立业务域，它会拒绝进入长链，强制要求先产出精简的**轻项目简介**并梳理依赖，避免 AI 被长上下文干到崩溃。
*   **🧠 `pm-mm`（纯粹血统的脑图发散）**
    *   **何时用：** 需要在一张图里看清业务主线。
    *   **机制：** **强规避页面约束**。系统被禁止在此输出页面名称和按钮交互，强制它只发散“业务流、角色参与节点和待核实风险”。

### 【骨架夯实队：奠定强契约锚点】
*   **📋 `pm-fl`（能力锚点：功能清单树 Feature List）**
    *   **何时用：** 脑图被你拍板后，要定出 v1.0 的排期范围界限。
    *   **机制：** 绝对下压到页面/能力级颗粒度：输出 `功能点 | 说明 | 高中低优先级 | 版本边界` 的表格。不写也不认具体操作步骤，它确立一切能力大盘。
*   **🏗️ `pm-ps`（页面锚点：页面承载矩阵 Page Structure）**
    *   **何时用：** 功能都有了，该装去哪个页面？
    *   **机制：** **只认 `pm-fl` 为“爹”**。它回答“这些能力放置点”，锁定`页面目标`及所属`关键区域`。不回答优先级，不可凭空制造不在上游清单的功能。

### 【交付可视队：提供消费端成品】
*   **📝 `pm-prd`（执行规格书）**
    *   **何时用：** 产出拿去给开发干活的重工业级文档。
    *   **机制：** 双重挂载于 FL 和 PS 上，继承涉及角色并划分边界。如果非常庞大，它自动启动“分块切片计划”，逐个模块生成再缝合。支持嵌入 Mermaid 数据流图。
*   **🎨 `pm-pt`（HTML代码级静态保真原型）**
    *   **何时用：** 需要看得见的布局交互界面。
    *   **机制：** 默认 Ant Design 企业后端画风，基于视觉基线复用。一切布局由 PRD 驱动生成。
*   **📌 `pm-pa`（悬浮式原型批注 Annotation）**
    *   **何时用：** 让开发不再痛苦地用双屏对齐 PRD 和 Figma。
    *   **机制：** 采用非侵入架构，在 HTML 原型旁打上角标（只支持点击拒绝乱闪烁），提取 PRD 内容悬浮展示，真正做到“画文合一”。

### 【质检监督员】
*   **⚖️ `pm-rv`（三体位审查 Review）**
    *   **何时用：** 觉得可以提测了，先让 AI 自己做审计。
    *   **机制：** 
        *   **安静自测**：核实 PRD 有没有造次“越权发散”未收录的页面。
        *   **拉会群打模式**：带上一句“拉个会吧”。系统立刻变身成测试、前后端、业务方五类人群，对当前产物分视角进行挑刺和定性拷问（Tiger级真坑 还是 Elephant级保留项）。

## 🕹 三、 六大典型实战剧本展示

为了让你更好地理解 ShitPM 如何在日常极其混乱的业务线中为你兜底，这里提供六个真实的 PM 工作场景剧本。

### 🎬 剧本一：老板的“一句话需求”（从 0 到 1 破局）
**场景背景：** 老板在群里发了一句：“我们在 App 首页加个签到领积分的功能吧，提高一下日活。下周要看设计稿。”
**实机操作流：**
1. **启动与圈定范围：** 新建聊天上下文，直接输入：`/pm-go 老板让我在首页加个签到日历，增加金币系统联动。`
2. **工作流拦截与定义：** AI 会判定这是一个全新的命题。它调用 `pm-scope`，不是直接一通乱写 PRD，而是先帮你输出**轻量级项目简介**，明确需要“签到模块”、“积分账户池”两个维度的支持，并进入 `project-status.json` 注册状态。
3. **推演脑图：** 接下来只需输入 `/pm-mm`。AI 将为你抛出一张 `Mermaid` 格式的思维导图（只写业务流血脉）：包括“用户签到 -> 判定连签 -> 积分派发 -> 防刷机制”。
4. **强行锁定边界：** 再次敲入 `/pm-fl`。AI 把脑图固化为了《V1.0 功能清单》。此时系统**不关心页面**，它帮你定下了：普通签到（高优）、补签卡（中优）、连续7天宝箱（高优）。这也是你的能力锚点。
5. **部署页面结构：** 敲入 `/pm-ps`。AI 把签到能力安放到了特定的页面容器，比如 `首页-顶部悬浮区`、`个人中心-日历弹层`。自此，所有发散结束。
6. **落笔生成 PRD：** 键入 `/pm-prd`。这是最爽的一步。AI 根据上面被锁死的锚点，洋洋洒洒生成几万字且**完全没有逻辑矛盾**的重工业 PRD。
7. **出概念原型：** 最后老板要看东西，输入 `/pm-pt`，拿走带气泡交互弹窗说明的原型 HTML 页面源码，发给前端。

### 🎬 剧本二：需求疯狂加塞，且自己手贱改了文档（状态修复）
**场景背景：** 周二晚上，业务方紧急要求：“那个签到必须分享到微信让新用户也能领红包！” PRD 甚至连功能清单都已经写好了。
**实机操作流：**
1. **直接“暴力美学”修改修改：** 你根本不想去重头对 AI 解释这套业务逻辑。你直接打开 `feature-list.md`，在 Markdown 原文里加入一行行功能表格。甚至顺手打开 `prd.md` 直接写了几句逻辑。
2. **AI 神级修补机制：** 保存文件后，唤起面板输入 `/pm-fix 帮我处理一下我的手动修改，业务方加了微信分享分享裂变的逻辑`。
3. **溯源与同步：** ShitPM 不会像普通 AI 那样把整篇文章全部重写。它像代码 diff 审查员一样分析：
   * “哦，老大在 FL 里加了 `能力边界：微信分享裂变`。”
   * “哦，那么在 PS 页面结构里，签到完成后的弹窗里得加个分享按钮。”
   * “好的，它自动修改下游所有关联的点，进行**静默回写**。” 所有节点数据同步刷新稳妥对接。

### 🎬 剧本三：跨多域的庞然大物（智能分块机制）
**场景背景：** 公司要重做整个“退款到退货物流闭环系统”，涉及前台、履约中台、物流顺丰接口三处。
**实机操作流：**
1. **立项：** 输入 `/pm-go 重构售后退换货系统，包含 C 端、商家端和寄件对接。`
2. **阻断器触发！：** ShitPM 分析并立刻亮红灯！在 `pm-scope` 阶段，AI 侦测到上下文爆炸的风险。**它绝不试图一口气吃成胖子**。
3. **规划子项目：** 它会停下来，给出建议：“主人，这个项目域太大了，极易产生生成幻觉。我建议拆分成三个子模块进行流水线作业：`模块A_用户端申请`、`模块B_商家工作台审核`、`模块C_物流回调监控`。”
4. **分块逐个击破：** 当你同意后，它利用 `pm-prd` 的切片计划能力，开始先专心攻克 `模块A`，生成后锁定。接着无缝衔接 `模块B`，并把 `模块A` 当成接口事实源引用。你最终得到的是一个架构严丝合缝的巨型系统集。

### 🎬 剧本四：面临历史陈旧屎山代码的接盘
**场景背景：** 你接手了一个离职老员工的烂摊子（一坨 Markdown 散乱摆着历史逻辑），现要在这个基础上优化“优惠券叠加工能”。
**实机操作流：**
1. **输入意图：** 敲 `/pm-go 我要在原有优惠券项目上重构叠加逻辑`。
2. **读作上下文，不作稳态锚点：** ShitPM 足够聪明。它侦测到目录下已有大量的历史物料，它会将这些物料加载为**参考依据（Reference）**而非新的起点。
3. **增量出单：** 在推演清单 `pm-fl` 时，它会主动对齐老文档，向你汇报：“以下是检测到的 V1 原有功能，本次仅针对结算环节进行修改。”。在写新的 `pm-prd` 时，它会标记 `【修订】` 和 `【新增】`，而不是重写一遍废话。

### 🎬 剧本五：被开发拷问前的预演（多体位 QA）
**场景背景：** 明天上午 10 点，你要带着刚出锅的新一版会员体系 PRD 去拉着技术评审。你怕自己被前端程序员喷代码逻辑不闭环。
**实机操作流：**
1. **生成之后，召唤评审团：** 刚刚利用 `/pm-prd` 写完主体内容后，直接敲入 `/pm-rv 拉个会吧，让技术专家团来挑挑刺。`
2. **角色扮演大乱斗：** 系统不仅会默默扫一次漏洞，随后立刻将自己的人格割裂：
   * **前端老李：** “在这个 `Page-03-收银台` 中，断网下的交互你没写降级策略，是弹 Toast 还是锁死按钮？”（Tiger 级漏洞，必须补）
   * **后端架构王博：** “你要求每次都拉取三方银行的积分汇率，如果是秒杀高并发这会垮掉。需要你确认是不是加 Redis 缓存或者允许异步写入？”
   * **业务线总监发哥：** “你的功能清单写了要支持积分退回，但是 PRD 里面没指明部分退款时，积分是全退还是等比例折算？”
3. **补坑：** 你看完了一身冷汗，赶紧拿他们提出的点针对性地打开 PRD 进行补充，然后在会上顺利推进无伤闪避。

### 🎬 剧本六：真正的画文合一无痛交接的悬浮组件
**场景背景：** 以前你用 Axure 或者墨刀画图，再用word写字，研发看东西必须痛苦地在两个页面之间切来切去。
**实机操作流：**
1. **生成原型闭环：** 用 `/pm-pt` 输出静态页面 `prototype.html` 后。
2. **植入悬浮批注：** 输入 `/pm-pa 给当前产生的 HTML 画一张热区补丁。`
3. **效果：** ShitPM 将会对 `prototype.html` 采取非侵入修改（不破坏原始布局结构），把 PRD 中的核心验收标准和各组件的判定逻辑，变成**悬浮提示角标**（Annotation）附着在 HTML 上。开发鼠标在按钮上一悬停，直接看到这块的具体 PRD 参数，享受沉浸式的实现过程。
