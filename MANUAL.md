# ShitPM 使用说明书

ShitPM 是一套面向 AI IDE 的 PM 工作流增强方案。

对普通用户来说，核心体验应该非常简单：**你只需要和 `pm-go` 说人话。**

你不用记脚本名，不用手动读状态文件，也不用自己判断现在该进哪个阶段。ShitPM 的目标是把“状态恢复、门禁判断、阶段路由、变更修复”都藏在 `pm-go` 背后，让你只关注业务问题本身。

## 一、现在怎么用

默认只用一种交互方式：

```text
/pm-go + 你的自然语言意图
```

比如：

- `/pm-go 现在做到哪个阶段了？`
- `/pm-go 当前还能不能继续推进？`
- `/pm-go 继续下一步。`
- `/pm-go 我刚手改了 PRD，帮我判断要不要回写。`

ShitPM 会在背后完成：

1. 恢复当前项目状态
2. 判断当前是否可推进
3. 识别你要查状态、继续推进、进入某阶段，还是做变更修复
4. 在需要推进时先做门禁判断
5. 能推进就继续，不能推进就直接告诉你缺什么

这里有一个很重要但不打断体验的前提：主交互方式仍然是 `pm-go + 自然语言`，但状态恢复、门禁判断和状态写入是否能被**真实完成**，仍取决于宿主集成或内部维护机制。如果当前宿主不能自动完成这些动作，系统会明确提示“当前结论是基于上下文或现有状态信息的判断”，并提醒你状态可能还需要确认或修复。只有当前轮存在可验证的执行证据时，系统才会把结论标记为“真实状态判断”。

## 二、你会看到什么样的回答

当你通过 `pm-go` 发起请求时，系统会尽量按固定结构回答：

1. 结论来源（真实状态判断 / 上下文推断）
2. 当前阶段
3. 当前是否可推进
4. 阻塞项或待确认项
5. 推荐下一步
6. 是否立即执行

其中“结论来源”不是装饰字段：它用来告诉你这轮结论到底来自真实执行，还是来自上下文推断。

也就是说，你不需要自己猜“现在是不是该做 PRD 了”“是不是该先评审”“是不是得先补功能清单”。系统会先告诉你当前局面，再决定要不要继续往下走。

## 三、四类高频场景怎么问

### 1. 我现在做到哪个阶段了？

直接问：

```text
/pm-go 现在做到哪个阶段了？
```

或者：

```text
/pm-go 帮我恢复一下当前项目状态。
```

你会得到：

- 当前阶段
- 上次完成了什么
- 当前锚点是什么
- 有没有阻塞项或待确认项
- 推荐下一步是什么

这类问题只会恢复状态，不会自动推进。

### 2. 我当前能不能继续？

直接问：

```text
/pm-go 当前能不能继续推进？
```

或者更具体一点：

```text
/pm-go 现在能不能直接进入 PRD？
```

系统会先在背后做门禁判断，然后告诉你两种结论之一：

- **可以推进**：说明可以进入下一阶段，并询问你是否立即执行
- **不能推进**：说明具体卡在哪里，例如缺少稳定锚点、还有阻塞项、还有待确认项没清

你不需要自己检查状态文件，也不需要自己跑校验命令。

### 3. 我想继续下一步

直接说：

```text
/pm-go 继续下一步。
```

或者：

```text
/pm-go 按推荐下一步继续推进。
```

系统会先恢复状态，再判断当前应该去哪里：

- 如果当前应该做脑图，就进入脑图阶段
- 如果当前应该做功能清单，就进入功能清单阶段
- 如果当前应该先评审，就进入评审阶段
- 如果当前还不能推进，就先告诉你为什么不能继续

也就是说，`pm-go` 不是只会“回答建议”，它也负责真正把你送到正确阶段。

### 4. 我刚手改了 PRD / 功能清单，帮我判断要不要回写

这是最典型的变更场景。

直接说：

```text
/pm-go 我刚手改了 PRD，帮我判断要不要回写。
```

或者：

```text
/pm-go 我刚改了功能清单，帮我判断影响范围和最小回写路径。
```

系统会优先识别为“变更修复”意图，并在背后进入 `pm-fix` 逻辑，判断这次修改属于哪一层：

- 只是 PRD 表达修订
- 影响页面承载
- 影响能力边界

然后告诉你：

- 需不需要回写
- 应该回写到哪一层
- 下游哪些产物需要同步
- 现在是否可以继续往下做

## 四、常见续做场景

### 场景A：隔了一两周，回来继续项目

直接说：

```text
/pm-go 继续上次项目，请先恢复当前状态并告诉我下一步。
```

系统会先恢复状态，再告诉你：

- 当前做到哪里
- 现在是否能继续
- 下一步建议做什么
- 要不要直接开始执行

### 场景B：PRD 做完后回头看脑图，发现上游错了

直接说：

```text
/pm-go 我在 PRD 阶段复盘脑图，发现这里有问题：xxx。请判断应该回退到哪一层，并给我最小修复路径。
```

系统会判断这是：

- 思路层问题
- 页面承载问题
- 能力边界问题

然后决定是留在当前层修，还是回退到更上游再重推下游。

### 场景C：原型已经出了，突然要加需求

直接说：

```text
/pm-go 原型已经出了，现在新增需求：xxx。请先评估影响范围，再告诉我应该怎么回写。
```

系统会先做影响分析，再告诉你：

- 只是改 PRD 和原型
- 还是要回到功能清单 / 页面结构重新锁边界

### 场景D：准备开评审会，想先让系统自查

直接说：

```text
/pm-go 当前内容准备评审，请先帮我检查现在能不能进评审，并在可以时直接进入评审阶段。
```

如果当前条件满足，系统就会进入评审；如果条件不满足，它会直接告诉你哪些问题要先补齐。

## 五、你不用再关心什么

对正常使用者来说，以下内容不需要作为主流程动作记忆：

- 状态文件怎么读
- 门禁脚本怎么跑
- 状态怎么 patch 写回
- 该进哪个子技能

这些都属于 ShitPM 的内部机制。你只需要把业务问题说清楚，让 `pm-go` 来做流程编排。

## 六、什么时候系统会拒绝继续

即使你只用自然语言，系统仍然不会“无脑往下写”。

以下情况会被拦住：

- 还有未解决的阻塞项
- 还有待确认项没有处理
- 上游稳定锚点缺失
- 版本关系未判清
- 当前并不适合直接进入目标阶段

这时系统不会强行推进，而是直接告诉你：

1. 为什么不能继续
2. 缺什么
3. 应该先补哪一步

## 七、一句话原则

**始终先找 `pm-go`，像和项目管家说话一样直接提需求；**  
**ShitPM 会在背后完成状态恢复、门禁判断、阶段路由和变更修复，并严谨地区分真实状态判断与上下文推断。**
