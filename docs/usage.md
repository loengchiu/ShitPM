# 使用说明

## 主入口

- `pm-go`

## 推荐口径

默认只通过 `pm-go` 交互，不要求用户手动记忆或单独调用其他子命令。

正常情况下，你可以把它理解成“只管说人话，剩下交给 `pm-go`”。  
但也要知道一条边界：这些结论是否来自**真实状态判断**，仍取决于当前宿主是否真的完成了状态恢复、门禁校验和状态维护，并且当前轮有可验证的执行证据；如果没有，系统应明确说明当前结论更接近基于上下文或现有状态信息的判断。

你可以直接这样说：

- `pm-go 现在做到哪个阶段了？`
- `pm-go 当前能不能继续？`
- `pm-go 继续下一步。`
- `pm-go 我刚手改了 PRD，帮我判断要不要回写。`

## 推荐用法

### 新需求

从 `pm-go` 开始，由系统优先结合当前状态和上下文：

- 调用内部技能 `pm-scope`
- 做范围判定
- 判断是否拆子项目
- 在范围已确认后，先进入分析阶段，沉淀项目简介、关键流程文字版和变更留痕
- 推进到后续 PM 阶段

### 现在做到哪了

调用：`pm-go 现在做到哪个阶段了？`

系统返回：

- 结论来源（真实状态判断 / 上下文推断）
- 当前阶段
- 当前锚点
- 阻塞项 / 待确认项
- 推荐下一步

### 当前能不能继续

调用：`pm-go 当前能不能继续推进？`

系统会先在背后做门禁判断，再告诉你：

- 结论来源（真实状态判断 / 上下文推断）
- 可以继续，是否立即执行
- 不能继续，原因是什么，还缺什么

### 继续下一步

调用：`pm-go 继续下一步。`

系统会先判断当前结论属于真实状态判断还是上下文推断，再决定是否进入下一阶段；如果当前可推进，则路由到对应能力。

### 人工改了 PRD 或功能清单

调用：`pm-go 我刚手改了 PRD/功能清单，帮我判断要不要回写。`

系统会在背后进入变更修复逻辑，判断：

- 结论来源（真实状态判断 / 上下文推断）
- 是否只影响当前层
- 是否要回写页面结构
- 是否要回写功能边界

## 最短测试路径

真实项目建议按下面顺序试：

1. `pm-go`
2. `pm-go 继续下一步`
3. `pm-go 继续下一步`
4. `pm-go 当前能不能继续推进？`
5. `pm-go 当前内容准备评审，请直接进入评审阶段`

详细测试步骤见：
- `docs/real-project-test.md`

## 手改 PRD 后的推荐做法

### 场景

你已经直接修改了 PRD，不想让系统继续引用旧版锚点，也不想让 AI 擅自重写上游。

### 做法

调用 `pm-go`，并直接说明你做了手工改稿、希望判断影响范围和最小回写路径。

### 目标

- 识别当前 PRD 是否已替代稳定锚点
- 判断修改只影响 PRD，还是影响页面清单或功能清单
- 给出回写建议，而不是直接全量重写
