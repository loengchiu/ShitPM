# 真实项目测试手册

本手册记录使用真实业务场景对 ShitPM 完整工作流进行端到端测试的步骤与预期结果。

## 前置条件

1. 完成安装：`powershell -File .\installers\install.ps1`
2. 在 IDE（GitHub Copilot / Codex / Cursor 等）中打开目标项目目录
3. 确认 `scripts\smoke-test.ps1` 通过（Exit Code 0）

## 测试场景：签到日历功能需求

以下用一个典型需求"在首页新增签到日历"完整跑通主链路。

### 第 1 步：启动主入口

在 AI 对话框输入：

```
/pm-go 老板让我在首页加一个签到日历，用户签到可以获得积分。
```

**预期**：AI 以 `pm-scope` 角色进行范围探索，输出轻量项目简介（Project Brief Lite），并暂停等待确认。

---

### 第 2 步：生成思维导图

确认简介后，输入：

```
/pm-mm
```

**预期**：输出结构化业务思维导图 Markdown，覆盖签到流程、积分逻辑、异常路径。

---

### 第 3 步：生成功能清单

```
/pm-fl
```

**预期**：输出 Feature List，包含签到触发、积分计算、历史记录、边界处理等条目，带优先级标注。

---

### 第 4 步：生成页面结构

```
/pm-ps
```

**预期**：基于 Feature List 输出页面与区域结构，不新增功能清单未覆盖的页面。

---

### 第 5 步：生成 PRD

```
/pm-prd
```

**预期**：生成完整 PRD，各章节与功能清单、页面结构双锚点对齐，无内容发散。

---

### 第 6 步：生成 HTML 原型

```
/pm-pt
```

**预期**：生成可在浏览器打开的高保真 HTML 原型，页面范围严格覆盖 PRD 所描述的功能集合。

---

### 第 7 步：添加原型批注

```
/pm-pa
```

**预期**：在原型 HTML 上渲染非侵入悬浮角标，点击后显示对应 PRD 条款。

---

### 第 8 步：评审

```
/pm-rv
```

**预期**：输出多角色审查清单，标注 Tiger（高风险）和 Elephant（隐藏复杂度）级别问题。

---

## 热更测试：pm-fix 验证

1. 手动修改 Feature List（如删除"历史查看"能力）
2. 输入：

```
/pm-fix
```

**预期**：AI 识别变更意图，自动追溯并更新 Page Structure 和 PRD 中的对应描述，保持产物咬合。

---

## 通过标准

- 每个环节产物不超出上游锚点范围
- 相邻环节之间无内容断层或矛盾
- `pm-fix` 能正确识别手动修改并回写上下游
# 真实项目测试手册

## 目标

用一个真实项目验证 `ShitPM` 当前主链是否可用：

`pm-go -> pm-fl -> pm-ps -> pm-prd -> pm-rv -> pm-pt -> pm-pa`

## 前提

- 已安装 `ShitPM`
- `installers/<host>/verify.ps1` 返回 `verify:ok`
- 项目目录中至少存在 `docs/` 目录

## 推荐测试顺序

### 1. 总入口测试

```text
请使用 pm-go。
项目目录：<project-root>
当前任务：这是一个新的 PM 需求。请先判断当前项目处于哪个阶段；如范围未清，请先做内置范围探索与拆分判定，并说明下一步应该进入哪个 PM 阶段。
```

通过标准：
- 能识别当前阶段
- 能说明本轮采用的锚点
- 能说明是否需要拆子项目
- 能推荐下一动作

### 2. 功能清单测试

```text
请使用 pm-fl。
项目目录：<project-root>
上游材料：<mindmap 或项目简介路径>
要求：
1. 输出页面/能力级颗粒度的功能清单。
2. 不要混入步骤级写法。
3. 每个模块标题下补“涉及角色”。
4. 主表只保留：#、功能点、说明、优先级、版本、PRD章节。
```

通过标准：
- 主表颗粒度正确
- 无步骤级功能点
- 模块级角色存在
- 优先级为中文

### 3. 页面清单测试

```text
请使用 pm-ps。
项目目录：<project-root>
上游功能清单：<feature-list 路径>
要求：
1. 只输出页面目标和页面结构树。
2. 每个区域固定写：对应功能点、作用、关键操作、关键状态。
3. 不要输出页面总表、结构稳定性结论、使用说明、自检。
```

通过标准：
- 页面目标清晰
- 区域结构完整
- 无过程性废话

### 4. PRD 测试

```text
请使用 pm-prd。
项目目录：<project-root>
上游功能清单：<feature-list 路径>
上游页面清单：<page-structure 路径>
要求：
1. 基于 Feature List + Page Structure 双锚点生成 PRD。
2. 如果规模过大，先给出 PRD 分块计划，再逐块写作。
3. 不要脑补 Feature List 和 Page Structure 中不存在的能力或页面事实。
4. 功能项必须包含：位置、目标、核心功能点、功能描述、界面承载说明。
```

通过标准：
- 来源受控
- 无上游之外的脑补
- PRD 在复杂场景下能先做分块计划

### 5. 评审测试

```text
请使用 pm-rv。
项目目录：<project-root>
当前功能清单：<feature-list 路径>
当前页面清单：<page-structure 路径>
当前 PRD：<prd 路径>
要求：
1. 检查三者是否冲突。
2. 判断问题属于能力边界、页面承载还是 PRD 规格层。
3. 如需回写，请明确回写到 pm-fl 还是 pm-ps。
```

通过标准：
- 能指出冲突点
- 能做回写判定
- 不会把所有问题都粗暴归到 PRD

### 6. 手改 PRD 后测试

```text
请使用 pm-fix。
项目目录：<project-root>
当前 PRD：<你手动修改后的 prd 路径>
要求：
1. 以我给出的当前 PRD 为主，不要回退引用旧版。
2. 判断这次修改只影响 PRD，还是影响 Page Structure 或 Feature List。
3. 如果当前 PRD 已替代旧版，请建议刷新 stable_baselines.prd。
```

通过标准：
- 优先采用显式 PRD 路径
- 能做版本关系判断
- 能做回写判定

### 7. Prototype 测试

```text
请使用 pm-pt。
项目目录：<project-root>
当前 PRD：<prd 路径>
要求：
1. 基于稳定 PRD 生成高保真 HTML 原型。
2. 如无视觉基线，先沉淀 visual-baseline.md。
3. 保持统一设计语言，不要自由发明另一套风格。
```

通过标准：
- 原型以 PRD 为事实源
- 生成或继承统一视觉基线
- 没有明显脱离 PRD 自由发挥

### 8. Prototype Annotation 测试

```text
请使用 pm-pa。
项目目录：<project-root>
当前 PRD：<prd 路径>
当前 Prototype：<prototype 路径>
要求：
1. 明确判定初始化标注或增量更新。
2. 生成带交互标注层的 HTML 原型。
3. 默认点击打开，不使用 hover 自动弹出。
```

通过标准：
- 角标和浮窗能覆盖关键模块
- 点击打开、X 关闭、拖拽和刷新同步规则成立
- 浮窗信息密度足够支撑研发理解

## 最低通过条件

满足以下 5 条即可判定 `ShitPM` 可进入真实项目试运行：

- 安装校验通过
- `pm-go` 能完成范围与阶段判定
- `pm-fl` 能稳定输出页面/能力级功能清单
- `pm-ps` 能稳定输出页面目标与页面结构树
- `pm-prd` 不再脱离双锚点自由发挥

## 常见失败信号

- `pm-go` 不识别当前阶段
- `pm-fl` 混入步骤级写法
- `pm-ps` 输出页面总表和说明性废话
- `pm-prd` 继续引用旧版锚点
- `pm-fix` 忽略显式给出的当前 PRD
