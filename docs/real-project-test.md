# 真实项目测试手册

本手册用于验证 `ShitPM` 在真实项目中的可用性。默认测试口径是：**普通用户只通过 `pm-go` 进行自然语言交互**。如需验证底层子技能，可在文末使用“维护者回归测试”。

## 前置条件

1. 已完成宿主安装。
2. 对应宿主的 `verify.ps1` 返回 `verify:ok`。
3. `scripts\smoke-test.ps1` 返回 Exit Code 0。
4. 项目目录中存在 `docs/` 目录。

## 测试场景：签到日历功能需求

以下用一个典型需求“在首页新增签到日历”验证主链路是否可跑通。

### 第 1 步：启动主入口

在 AI 对话框输入：

```text
/pm-go 老板让我在首页加一个签到日历，用户签到可以获得积分。请先判断当前阶段；如果范围还不清，就先做范围探索并告诉我下一步。
```

通过标准：

- 能识别当前项目属于新需求起点
- 能先进入范围判断，而不是直接输出 PRD
- 能给出“当前阶段 / 当前是否可推进 / 推荐下一步”

### 第 2 步：继续到脑图或功能边界阶段

在 AI 对话框输入：

```text
/pm-go 继续下一步。
```

通过标准：

- 能基于当前状态决定是否进入 `pm-mm`
- 若仍缺范围确认，能明确说明为什么不能继续
- 若可以继续，输出业务流程、角色、关键判断节点和风险

### 第 3 步：进入功能清单阶段

在 AI 对话框输入：

```text
/pm-go 如果当前已经适合进入功能清单阶段，就继续推进；否则告诉我还缺什么。
```

通过标准：

- 能在合适时进入 `pm-fl`
- 功能清单保持页面/能力级颗粒度
- 不混入步骤级写法
- 明确优先级和版本边界

### 第 4 步：进入页面结构阶段

在 AI 对话框输入：

```text
/pm-go 如果功能边界已经稳定，请进入页面结构阶段。
```

通过标准：

- 能在合适时进入 `pm-ps`
- 页面结构只承载功能清单已有能力
- 不凭空新增页面能力

### 第 5 步：进入 PRD 阶段

在 AI 对话框输入：

```text
/pm-go 如果当前已经满足条件，请进入 PRD 阶段；否则告诉我为什么还不能继续。
```

通过标准：

- 能在合适时进入 `pm-prd`
- PRD 不脱离 Feature List 和 Page Structure 双锚点
- 复杂场景下能先给出分块计划

### 第 6 步：进入评审阶段

在 AI 对话框输入：

```text
/pm-go 当前内容准备评审，请先判断能否进入评审，并在可以时直接进入评审阶段。
```

通过标准：

- 能在合适时进入 `pm-rv`
- 能识别能力边界、页面承载、PRD 规格层问题
- 能输出回写建议

### 第 7 步：手改后回写测试

手动修改 PRD 或 Feature List 后，在 AI 对话框输入：

```text
/pm-go 我刚手改了 PRD，帮我判断要不要回写，以及最小回写路径是什么。
```

通过标准：

- 能优先识别为变更修复场景
- 能判断只影响当前层，还是要回写到 `pm-ps` / `pm-fl`
- 能说明当前结论来自真实状态判断还是上下文推断

### 第 8 步：进入 Prototype 阶段

在 AI 对话框输入：

```text
/pm-go 如果当前 PRD 已稳定，请进入原型输出阶段；否则告诉我还缺什么。
```

通过标准：

- 能在条件满足时进入 `pm-pt`
- 原型内容不脱离 PRD
- 视觉风格保持统一

### 第 9 步：进入批注阶段

在 AI 对话框输入：

```text
/pm-go 如果当前原型和 PRD 已满足条件，请进入原型批注阶段。
```

通过标准：

- 能在条件满足时进入 `pm-pa`
- 批注层覆盖关键模块
- 标注交互规则清楚可用

## 最低通过条件

满足以下条件，即可认为 `ShitPM` 可以进入真实项目试运行：

- `pm-go` 能完成范围与阶段判定
- `pm-go` 能在“继续下一步”场景下稳定路由
- `pm-go` 能在不满足条件时明确阻止推进
- `pm-prd` 不再脱离双锚点自由发挥
- `pm-rv` 能输出有效评审问题
- `pm-fix` 能做最小回写判定

## 常见失败信号

- `pm-go` 不能识别当前阶段
- `pm-go` 在前提不足时仍强行推进
- `pm-fl` 混入步骤级写法
- `pm-ps` 输出超出功能清单范围的页面事实
- `pm-prd` 继续引用旧版锚点
- `pm-fix` 忽略显式给出的当前 PRD
