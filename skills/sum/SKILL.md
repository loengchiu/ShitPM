# PM Analysis

## 作用

把范围澄清后的材料整理成可进入思维导图阶段的项目简报。这个阶段不直接写 PRD，而是先形成稳定的中间事实层，供后续 `mind`、`feat` 和 `prd` 复用。

使用以下参考文件：

- `../../contracts/artifact-schema.md`
- `../../contracts/done-criteria.md`
- `../../contracts/error-handling.md`
- `../../templates/project-brief-lite.md`

## 输入

按以下优先级读取：
1. 当前项目目录与已有上下文
2. 范围澄清结论
3. 来源材料索引
4. 会议纪要
5. 原始需求点、补充说明与参考材料
6. 如存在，二次访谈或补充澄清信息
7. 当前稳定锚点与最新产物

## 输入材料预处理

当来源材料，如操作手册、功能清单、需求文档、会议纪要等，预估超过 3000 字时，不得直接将全文作为分析输入。

1. `检查预处理缓存`：先检查 `docs/cache/material-extract.md` 是否已存在，由 `scope` 阶段生成
2. `若已存在`：
   - 读取缓存顶部的元数据，确认提取来源、是否存在待分类内容、是否允许定向回查原文
   - 直接使用提取结果作为来源材料，不再读取原始材料全文
   - 若缓存标注“存在待分类内容”，在分析完成后检查待分类段落是否影响关键结论，必要时定向回查原文相关片段
3. `若不存在`：
   - 执行与 `scope` 相同的结构化提取流程
   - 将提取结果写入 `docs/cache/material-extract.md`，并补齐相同的元数据
   - 后续分析只基于提取结果
4. `提取维度`：业务目标与背景、涉及角色及其职责、业务流程与关键操作结果（提取为流程描述，不保留纯界面操作细节）、业务规则与约束条件、数据实体与关键字段、异常场景与边界条件
5. `提取原则`：丢弃纯界面表现，保留承载业务规则的入口、触发条件、权限线索和状态条件；提取结果控制在 2000 字以内，若仍超长，则在 `docs/cache/material-extract.md` 内按业务模块拆分为多个提取小节
6. `缓存边界`：`docs/cache/material-extract.md` 仅作为上下文预处理缓存，不属于正式产物，不写入 `stable_baselines`
7. `短材料免处理`：若材料预估不超过 3000 字，跳过预处理，直接读取全文

## 输出

将项目简报写入工作项目的 `docs/briefs/` 目录。

## 规则

- 先形成项目简报，再进入思维导图。
- 项目简报必须按模板覆盖：背景、资料分析结论、范围、角色、关键流程文字版、关键规则、风险与待确认项、变更与修正留痕。
- `资料分析结论` 必须区分：已确认事实、高可信推断、冲突或待核实信息。
- 如存在二次访谈、补充澄清或理解修正，写入 `二次访谈补充与修正`；如不存在，可留空或省略内容，但不要伪造。
- `变更与修正留痕` 统一记录需求变更、理解修正和补充访谈带来的结论变化。
- 项目简报聚焦“项目总览 + 资料分析结论 + 关键流程文字版 + 留痕”，不要扩写成第二份 PRD。
- 关键流程采用文字描述，不要求本阶段补流程图。
- 信息不足时显式标记待确认，不擅自补全。
- 如果核心问题定义本身冲突、边界互相打架，先标记为风险并停在本阶段。
- 只有当项目简报足以支撑 `mind` 时，才允许交接下游。

## 不负责

- 不直接生成 PRD
- 不输出页面结构
- 不设计原型
- 不替代思维导图阶段

## 方法论：资料分析框架

整理项目简报时，确保以下 5 个维度都已覆盖，信息已足够时可跳过：

1. `业务目标`：本次需求的业务目标是什么？如存在明确指标，可记录指标；如没有，则记录核心交付目标或关键改进点。
2. `当前痛点`：用户当前遇到什么问题？影响范围多大？
3. `涉及角色`：所有涉及角色是谁？各自的核心诉求是什么？
4. `现有流程`：当前业务流程是什么？关键瓶颈在哪里？
5. `约束条件`：有什么技术、合规、资源方面的约束？

使用原则：

- 不是每次都要覆盖全部 5 个维度，信息已足够时直接跳过
- 如果某个维度信息不足，显式标记为待确认，不擅自补全
- 分析结论必须区分：已确认事实、高可信推断、冲突或待核实信息

## 完成自审

项目简报生成完毕后，逐条检查以下项目，不通过的必须修正：

1. `事实分层`：资料分析结论是否明确区分了已确认事实、高可信推断、冲突或待核实信息？
2. `关键流程`：关键流程是否可执行？能否从文字描述中看出谁在什么条件下做了什么？
3. `角色完整`：所有涉及角色是否列出？是否存在未知角色或未标注角色的功能？
4. `待确认项`：每个待确认项是否标注了谁确认和什么时候确认？是否存在无法推进的待确认项？
5. `范围边界`：能否一句话说清做什么和不做什么？是否存在范围互相打架的情况？

执行规则：

- 自审发现的问题先直接修正
- 若问题仅是表述不清，留在当前稿件内修正
- 若问题说明上游范围澄清本身不稳定或缺失关键信息，停止推进并回退到 `scope`
- 全部通过后，才允许进入状态更新

## 状态更新

- 进入本阶段时，将当前阶段设为 `sum`
- 更新 `last_action`
- 更新 `next_recommended` 为 `mind`
- 更新 `context_summary`
- 将本轮项目简报写入 `latest_artifacts.briefs`
- 将仍影响推进的待确认项写入 `pending_confirmations`

## 完成定义

- 已形成可进入思维导图阶段的项目简报
- 简报已覆盖背景、资料分析结论、范围、角色、关键流程、关键规则、风险与待确认项、变更与修正留痕
- 如存在二次访谈或补充修正，已写入对应章节
- 已区分已确认事实、高可信推断和冲突/待核实信息
- 未扩写成第二份 PRD
- 产物足以支撑后续 `mind`

## 错误处理与回退

- 若范围澄清结果不足以支撑分析，回退到 `scope`
- 若来源材料不足以整理关键流程、关键规则或角色边界，停止推进并要求补充材料
- 若补充访谈导致原理解发生明显冲突，先修订项目简报，再继续下游
- 若分析过程中发现核心问题定义冲突，明确记录风险并停止继续下游

## 交接

- 将项目简报交给 `mind`
- 若分析结果仍不足以支撑思维导图，继续补分析，不直接交接
