# PM Analysis

## 作用

把范围澄清后的材料整理成可进入思维导图阶段的项目简报。这个阶段不直接写 PRD，而是先形成稳定的中间事实层，供后续 `mind`、`feat` 和 `prd` 复用。

使用以下参考文件：

- `../../contracts/artifact-schema.md`
- `../../contracts/done-criteria.md`
- `../../contracts/error-handling.md`
- `../../templates/project-brief-lite.md`

## 输入

按以下优先级读取：
1. 当前项目目录与已有上下文
2. 范围澄清结论
3. 来源材料索引
4. 会议纪要
5. 原始需求点、补充说明与参考材料
6. 如存在，二次访谈或补充澄清信息
7. 当前稳定锚点与最新产物

## 输出

将项目简报写入工作项目的 `docs/briefs/` 目录。

## 规则

- 先形成项目简报，再进入思维导图。
- 项目简报必须按模板覆盖：背景、资料分析结论、范围、角色、关键流程文字版、关键规则、风险与待确认项、变更与修正留痕。
- `资料分析结论` 必须区分：已确认事实、高可信推断、冲突或待核实信息。
- 如存在二次访谈、补充澄清或理解修正，写入 `二次访谈补充与修正`；如不存在，可留空或省略内容，但不要伪造。
- `变更与修正留痕` 统一记录需求变更、理解修正和补充访谈带来的结论变化。
- 项目简报聚焦“项目总览 + 资料分析结论 + 关键流程文字版 + 留痕”，不要扩写成第二份 PRD。
- 关键流程采用文字描述，不要求本阶段补流程图。
- 信息不足时显式标记待确认，不擅自补全。
- 如果核心问题定义本身冲突、边界互相打架，先标记为风险并停在本阶段。
- 只有当项目简报足以支撑 `mind` 时，才允许交接下游。

## 不负责

- 不直接生成 PRD
- 不输出页面结构
- 不设计原型
- 不替代思维导图阶段

## 状态更新

- 进入本阶段时，将当前阶段设为 `sum`
- 更新 `last_action`
- 更新 `next_recommended` 为 `mind`
- 更新 `context_summary`
- 将本轮项目简报写入 `latest_artifacts.briefs`
- 将仍影响推进的待确认项写入 `pending_confirmations`

## 完成定义

- 已形成可进入思维导图阶段的项目简报
- 简报已覆盖背景、资料分析结论、范围、角色、关键流程、关键规则、风险与待确认项、变更与修正留痕
- 如存在二次访谈或补充修正，已写入对应章节
- 已区分已确认事实、高可信推断和冲突/待核实信息
- 未扩写成第二份 PRD
- 产物足以支撑后续 `mind`

## 错误处理与回退

- 若范围澄清结果不足以支撑分析，回退到 `scope`
- 若来源材料不足以整理关键流程、关键规则或角色边界，停止推进并要求补充材料
- 若补充访谈导致原理解发生明显冲突，先修订项目简报，再继续下游
- 若分析过程中发现核心问题定义冲突，明确记录风险并停止继续下游

## 交接

- 将项目简报交给 `mind`
- 若分析结果仍不足以支撑思维导图，继续补分析，不直接交接
