# PM Scope

## 目标

在主链开始前完成最小必要的范围探索和拆分判定，避免直接进入后续产物生成。

## Hard Gate

在 `scope` 完成并得到用户确认前：

- 不进入 `mind` / `feat` / `page` / `prd`
- 不产出下游正式产物
- 不把未确认的拆分结论当既定范围

## 输入材料预处理

当用户提供的材料，如操作手册、功能清单、需求文档、会议纪要等，预估超过 3000 字时，不得直接将全文作为分析输入，必须先执行预处理。

### 预处理流程

1. `评估规模`：读取材料前，先评估材料总字数，可通过文件大小或目录结构判断
2. `结构化提取`：从原始材料中提取与以下维度相关的信息：
   - 业务目标与背景
   - 涉及角色及其职责
   - 业务流程与操作步骤，提取为流程描述，不保留纯界面操作细节
   - 业务规则与约束条件
   - 数据实体与关键字段
   - 异常场景与边界条件
3. `输出预处理缓存`：将提取结果写入 `docs/cache/material-extract.md`，作为后续阶段的实际输入
4. `默认不再全文回读`：预处理完成后，后续阶段默认只读取提取结果；如需确认具体细节，允许定向回查原始材料中的相关片段，但不得将原始材料全文重新载入上下文

### 提取原则

- 丢弃纯界面表现，如按钮位置、弹窗样式、页面布局、颜色图标等；但保留承载业务规则的入口、触发条件、权限线索和状态条件
- 只提取规则和约束，不提取操作步骤细节，如点击哪里、输入什么
- 提取结果控制在 2000 字以内；若仍超长，在 `docs/cache/material-extract.md` 内按业务模块拆分为多个提取小节
- 若材料中存在无法判断是否相关的内容，保留到提取结果的 `待分类` 段落，不直接丢弃
- 提取完成后，告知用户：“已从原始材料中提取 X 条业务事实，预处理缓存见 docs/cache/material-extract.md”

### 预处理缓存元数据

提取结果文件顶部必须包含以下元数据：

- `提取来源`：[原始材料文件名或描述]
- `是否存在待分类内容`：是 / 否
- `是否允许后续定向回查原文`：是 / 否
- `提取时间`：[日期]

### 缓存边界

- `docs/cache/material-extract.md` 仅作为上下文预处理缓存，不属于正式产物
- 该缓存不写入 `stable_baselines`
- 该缓存不作为正式交付件或事实源锚点

### 短材料免处理

若材料预估不超过 3000 字，跳过预处理，直接读取全文。

## 执行顺序

1. 读取现有上下文
2. 先判断任务是否明显过大、是否应拆分
3. 信息不足时逐轮澄清
4. 形成范围结论或拆分方案
5. 用户确认后返回统一路由入口

## 负责内容

- 澄清当前任务目标
- 判断是否属于单一 PM 任务
- 判断是否需要拆成多个子项目
- 形成轻量项目简介所需的基础信息

## 交互原则

- 先看上下文，再提问
- 一次只问一个问题
- 优先问当前最影响范围边界或拆分判断的问题
- 信息已足够时停止追问，转为确认

## 拆分优先原则

若需求同时覆盖多个相对独立的业务域、流程闭环或交付模块，优先拆分，不在一个大任务里继续细化全部子域。

拆分时至少给出：

- 拆分方案
- 每个子项目的目标与边界
- 依赖关系与推荐顺序
- 当前建议优先推进的子项目

不需要完整待办清单；只保留能支撑主链推进的轻量拆分记录。

## 输出要求

至少明确：

- 当前任务目标
- 范围边界
- 是否需要拆子项目
- 如拆分，子项目名称、目标、依赖、顺序
- 当前优先进入主链的子项目
- 如缺少基础总览，补齐轻量项目简介字段
- 推荐返回统一路由入口后的下一动作

若仍在澄清中，还要明确：

- 当前尚未确认的关键问题
- 这些问题为什么影响范围判断
- 下一轮只准备确认哪一个问题

## 下一步回流

`scope` 不是终点。执行完成后必须把控制权返回统一路由入口，由路由入口继续：

- 记录范围与拆分结果
- 判断当前阶段
- 决定进入分析、思维导图或其他下游阶段

只有在用户确认当前范围结论或拆分方案后，才能继续主链。

## 方法论：问题框架

当用户需求模糊、范围不清时，按以下结构化问题逐项澄清，一次只问一个问题，信息已足够时跳过：

1. `业务目标`：你希望达成什么业务目标？如存在明确指标，可一并说明；如没有，则说明核心交付目标或关键改进点。
2. `用户角色`：谁会使用这个功能？不同角色有什么不同的需求？
3. `使用场景`：用户在什么情况下使用？触发使用的条件是什么？
4. `约束条件`：有什么技术、时间、资源、合规方面的约束？
5. `成功标准`：怎么判断这件事做成了？如无明确业务指标，则明确验收口径、交付目标或关键改进点。

使用原则：

- 按阻塞度从高到低选择当前最该问的 1 个问题
- 不是每次都要问完 5 个，信息已足够时直接跳过后续维度
- 用户回答后，先检查是否存在矛盾、遗漏或范围冲突，再决定是否继续追问
- 澄清完毕后，将结论写入范围结论，不保留问答过程

- `latest_artifacts.briefs`（如本轮补齐了轻量项目简介）
- `blockers`
- `pending_confirmations`
