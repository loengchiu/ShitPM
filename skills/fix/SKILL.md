# PM FIX

## 目标

在用户手动修改已确认产物后，不直接重跑整条 PM 链路，而是先做影响范围判定、版本关系判定和回写处理。

如果本轮自动改动了其他生成物，必须同步留下改动记录，避免后续无法追溯“为什么改、改了谁、改了哪些上游事实源”。

## 输入优先级

1. 用户显式给出的当前变更产物路径
2. `docs/project-status.json` 中与该产物对应的稳定基线
3. 项目目录扫描得到的候选产物，结合命名、版本信息、修改时间和当前上下文判定出的当前工作稿；若仍无法判清，不自动认定，停止推进并要求确认

## 必做判定

### 1. 当前变更产物与稳定基线的关系

- 当前变更产物是否已经替代对应的 `stable_baselines`
- 是否存在新版本已生成但稳定锚点仍指向旧版

### 2. 修改层级判定

- 仅文档表达修订
- 业务流程 / 结构层修订
- 页面承载层修订
- 能力边界层修订

### 3. 回写判定

- 仅表达层：停留在当前产物
- 影响业务流程、判断节点、状态分支或系统结构：回写 `mind`
- 影响页面承载：回写 `page`
- 影响能力边界：回写 `feat`，必要时继续影响 `mind` 与 `page`

## 影响分析

当确认需要回写时，必须先完成影响分析，再决定回写范围：

1. `变更提取`
   - 精确列出本次修改了什么：字段定义 / 校验规则 / 业务流程 / 页面元素 / 能力边界 / 纯文案
2. `传播路径判定`
   - 字段定义或校验规则变更：检查 `mind / feat / page` 是否存在对应字段线索，有则回写
   - 业务流程、判断节点、状态分支变更：回写 `mind`
   - 页面承载变更：回写 `page`
   - 能力边界变更：回写 `feat`，并检查是否连锁影响 `mind / page`
3. `影响范围枚举`
   - 列出所有需要检查的产物与章节，不遗漏直接受影响项
4. `影响程度分级`
   - `局部`：只影响单个模块、章节或页面
   - `结构性`：影响模块间关系、排序、分组、全局规则或整体结构
5. `影响分析报告`
   - 输出本次变更的影响分析摘要，作为回写执行的前置依据

## 回写粒度判定

- `局部`：只改对应章节或模块，不动其他部分；修正记录中必须标注具体改动范围
- `结构性`：必须全量回写该产物，不允许只改片段
- 若改动涉及模块间关系、排序、分组或全局性规则，直接视为 `结构性`
- 部分回写后，仍需补跑对应产物 skill 的自审部分，确认局部改动未破坏整体一致性

## 回写冲突处理

当回写目标产物与当前稳定锚点不一致时，按以下规则处理：

1. `锚点未确认`
   - 允许覆盖并更新锚点
2. `锚点已确认 + 来源明确为评审驱动`
   - 允许覆盖并更新锚点，并在修正记录中标注“评审驱动变更”
3. `锚点已确认 + 来源明确为用户手动修改`
   - 标记为冲突，不自动覆盖
   - 在修正记录中列出冲突内容，要求用户确认后再执行回写
4. `锚点已确认 + 来源不明`
   - 标记“来源待确认”，不自动假设为用户手动修改
   - 在修正记录中说明待确认来源，再决定是否覆盖
5. `连锁冲突`
   - 回写 A 导致 B 也需要改，且 B 的锚点也已确认时，逐层标记冲突，不自动级联覆盖
   - 所有冲突统一收敛到用户确认

## 输出要求

必须明确输出：

- 本轮实际采用的当前变更产物路径
- 当前变更产物与对应稳定基线的关系
- 修改属于哪一层
- 是否需要回写 `mind`
- 是否需要回写 `page`
- 是否需要回写 `feat`
- 是否需要刷新对应 `stable_baselines`
- 推荐下一动作

如本轮已自动改动其他生成物，还必须输出一份修正记录，写入 `docs/fix-records/`。  
结构参考：`../../templates/fix-record.md`

修正记录至少包含：

- 本轮实际采用的当前变更产物路径
- 当前变更产物与对应稳定基线的关系
- 影响层级
- 影响分析摘要
- 回写粒度（局部 / 结构性）
- 冲突来源（评审驱动 / 用户手动修改 / 来源待确认 / 无）
- 本轮自动改动了哪些生成物
- 每个生成物的改动原因、改动范围、改动后路径
- 是否刷新了对应 `stable_baselines`
- 哪些事项仍需人工确认

## 状态更新

- 如当前变更产物已替代旧版，刷新对应 `stable_baselines`
- 如需回写，写入 `pending_confirmations` 或 `blockers`
- 如本轮已生成修正记录，更新 `latest_artifacts.fix_records`

## 禁止事项

- 不得在未判清版本关系前直接引用旧稿
- 不得在未判清影响层级前直接重写上游
- 不得自动改动其他生成物却不留修正记录
- 不得把影响业务流程、判断节点或结构的改动误判成纯文案问题

## 前置检查

执行前先读取 `docs/project-status.json`。
1. 如果文件不存在，停止执行并告知用户："当前项目尚未初始化 ShitPM，请先输入'初始化 ShitPM'。"
2. 检查 `blockers`：
   - 若 `blockers` 指向的正是当前 `fix` 目标产物，允许继续；`fix` 本身就是为解决它而来
   - 若 `blockers` 指向其他产物，停止并告知用户
3. 检查 `pending_confirmations`：
   - 若 `pending_confirmations` 指向的正是当前 `fix` 目标产物，允许继续
   - 若 `pending_confirmations` 指向其他产物，停止并告知用户
4. 若未满足，不得继续推进。
