# PM PRD

## 作用

生成和维护 PRD。PRD 是后续 Prototype 与评审的主规格锚点。

## 输入

按以下优先级读取：
1. 稳定 Feature List
2. 稳定 Page Structure
3. 影响范围、优先级、规则表述或验收口径的补充澄清信息
4. 历史 cards / interviews / 旧稿（仅作补充参考，不得覆盖当前稳定锚点）

如存在 `docs/project-status.json`，优先继承：

- `stable_baselines.feature_list`
- `stable_baselines.page_structure`

## 输出

将 PRD 写入工作项目的 `docs/prd/` 目录。  
使用 `../../templates/prd.md` 作为强约束交付模板。

## 规则

- 内容来源优先级固定为：`Feature List + Page Structure > 补充澄清信息 > 历史材料`
- 历史材料只能补充，不能覆盖当前稳定锚点。
- `上游锚点` 只填写本轮直接支撑 PRD 的当前稳定事实源，默认应为当前 `Feature List` 与当前 `Page Structure`。
- `来源材料` 只填写本轮真实直接使用的补充材料，并说明用途。
- 若工作项目中同时存在多版 Feature List / Page Structure，只能继承 `stable_baselines` 指向的当前稳定版，不得自行回退引用旧版。
- 必须严格按模板主骨架输出，不得自由重组主章节。
- 信息不足时可在对应章节显式标记“待确认”，不得删除或改写模板主骨架。
- 使用纯中文表达，少量行业通用英文缩写除外。
- 只描述业务规则、状态、数据流向和前端表现，不干涉底层技术实现。

#### 按页面组织

- 第 5 章按页面组织，每个页面对应 Page Structure 中的一个页面
- 页面顺序与 Page Structure 中的定义一致
- 每个页面章节顶部标注：页面结构引用（PS-__）和关联功能点（FL-__）
- 页面内按区域描述，区域定义来自 Page Structure
- 区域内的功能点引用功能清单编号
- 不允许出现 Page Structure 中不存在的页面或区域
- 不允许出现功能清单中不存在的功能点

#### 关键场景覆盖要求

按复杂度灵活组织，但必须覆盖真实存在的关键场景。以下 9 类场景只要真实存在就必须写，不存在的不要硬凑：

1. **展示**：数据怎么展示、展示格式、排序规则、空状态展示内容
2. **操作**：按钮/链接的触发条件和执行结果（点击后系统做什么、用户看到什么）
3. **输入**：输入方式、校验规则、默认值、格式限制、输入长度
4. **加载**：首次加载、分页加载、搜索加载、加载中状态（loading/骨架屏）、空数据状态、加载失败状态
5. **弹窗**：弹窗/抽屉/确认框的触发条件、弹窗内容、操作项、关闭行为（点取消、点遮罩层、按 Esc）
6. **异常**：权限不足时的提示和跳转、数据异常（必填为空、格式错误）、网络异常、超时、并发冲突
7. **数据规范**：字段展示格式（日期格式、金额精度、百分比）、单位、脱敏规则、长文本截断
8. **交互逻辑**：按复杂度分层写（见下方详略规则），作为功能描述的一部分，不单独成章
9. **状态流转与边界场景**：状态变更的触发条件和执行动作、边界值（最大/最小/为零）、临界状态

#### 交互逻辑分层写法

交互逻辑不是独立章节，而是功能描述的一部分。按业务复杂度决定详略：

- **简单交互**（如：点击删除、勾选复选框）：1-2 句写清触发、反馈和结果
  - 格式：触发 → 系统响应 → 用户可见结果
  - 示例：点击"删除"按钮 → 弹出确认框"确定删除该记录？" → 确认后列表移除该行，提示"已删除"
- **中等复杂交互**（如：带校验的表单提交、多步操作）：补充交互顺序、关键状态变化和必要分支
  - 格式：步骤1 → 步骤2 → 正常路径结果 / 异常分支结果
  - 示例：填写表单 → 点击"提交" → 校验通过：状态变为"待审批"，按钮置灰，通知审批人 / 校验失败：标红错误字段，滚动到第一个错误位置
- **高复杂交互**（如：审批流、多方协作、状态机驱动）：细写触发条件、交互顺序、反馈方式、状态变化、关键分支和中断条件
  - 格式：前置条件 → 触发 → 步骤1 → 步骤2 → ... → 正常结果 / 分支A / 分支B / 中断处理
  - 示例：审批人打开审批详情 → 查看申请信息和历史记录 → 点击"通过" → 系统校验当前状态为"待审批" → 通过：状态变为"已通过"，触发建档任务，通知申请人 / 拒绝：弹出必填拒绝原因输入框 → 提交后状态变为"已拒绝"，通知申请人

判断标准：
- 只有一个触发和一个结果 → 简单
- 有校验或 2-3 个顺序步骤 → 中等
- 有状态机、多方协作、多个分支或中断条件 → 高复杂

#### 禁用模糊表述

以下表述在 PRD 中禁止使用，每条描述必须写清楚具体的触发条件、系统行为和用户可见结果：

- ❌ "需支持XXX" → ✅ 写清楚支持什么、什么条件下触发、系统做什么
- ❌ "需考虑XXX" → ✅ 写清楚考虑的具体规则和处理方式
- ❌ "按规范处理" → ✅ 写清楚具体是什么规范、怎么处理
- ❌ "详见原型" → ✅ 把原型中的关键信息用文字描述出来
- ❌ "同常规" → ✅ 写清楚具体是什么常规做法
- ❌ "待定" → ✅ 写清楚当前假设是什么，标注需要谁确认

#### 边界场景要求

- 每个涉及数值输入的区域必须标注边界值（最大值、最小值、精度）
- 每个涉及列表的区域必须标注空状态、数据量上限（如有）
- 每个涉及状态流转的功能必须标注临界状态（如：审批中能否编辑、删除后状态怎么变）
- 每个涉及并发操作的功能必须标注冲突处理方式

#### 长文档分块生成协议

当 PRD 预估超过 1000 行时，必须启用分块生成模式。

生成顺序：
1. 数据字典（第 4 章）
2. 横切规则（第 6 章）
3. 权限矩阵（第 3 章）
4. 页面说明（第 5 章），逐页生成
5. 验收标准汇总（第 9 章）
6. 文档概述 + 范围（第 1-2 章）

分块生成循环（每个页面）：
1. 回读上游锚点中与当前页面相关的内容
   - 读取 Feature List 中当前页面关联的功能点（只读相关条目）
   - 读取 Page Structure 中当前页面的区域定义和元素清单
   - 读取数据字典中当前页面涉及的字段
2. 回读已生成页面的结构化摘要（不读全文）
3. 读取当前页面的 PRD 模板结构
4. 生成当前页面（每个区域按 9 类场景检查覆盖）
5. 保存到文件
6. 生成当前页面的结构化摘要
7. 从上下文中移除当前页面的完整内容，只保留摘要

摘要格式：
### PS-__ [页面名称]
- 关联功能点：FL-__、FL-__
- 区域：[区域列表]
- 涉及字段：[数量] 个，特殊规则：[关键规则]
- 待确认：[如有]
- 验收标准：[数量] 条

#### 一致性校验

全部页面生成完毕后，必须执行独立的一致性校验。

校验清单：
- 功能覆盖：Feature List 中每条"本期"功能点是否都有对应描述
- 页面覆盖：Page Structure 中每个页面是否都有对应章节
- 字段一致性：同一字段在不同页面的展示格式和校验规则是否一致
- 权限一致性：权限矩阵与各页面按钮可见性是否匹配
- 状态一致性：状态流转图与各页面状态筛选/展示是否一致
- 交叉引用完整性：所有 FL-__、PS-__、DD-__ 引用是否指向真实条目
- 场景覆盖：每个区域的 9 类场景是否覆盖了真实存在的关键场景（不存在的类别允许省略）
- 模糊表述检查：全文搜索"需支持"、"需考虑"、"按规范处理"、"详见原型"、"同常规"、"待定"，确保零命中

执行方式：
- 独立上下文，只读取 PRD 全文 + 功能清单 + 页面结构 + 数据字典
- 不做生成，只做检查
- 输出问题清单，标注位置和修复建议

## 业务属性继承

PRD 不得丢失上游业务属性：

- `Feature List` 中的角色、范围与关键业务边界必须在 PRD 中保留
- `Page Structure` 中的页面与区域承载关系必须在 PRD 中可回溯
- 当优先级直接影响本期范围、实现顺序或交付边界时，必须在 PRD 中体现

## 详细需求说明规则

每个模块下的子功能至少包含：

- 位置
- 目标
- 核心功能点
- 功能描述

### 功能描述

按复杂度灵活组织，但必须覆盖真实存在的关键场景，必要时包括：

- 展示
- 操作
- 输入
- 加载
- 弹窗
- 异常
- 数据规范
- 交互逻辑
- 状态流转与边界场景

不允许只写“需支持”“需考虑”“按规范处理”。

其中“交互逻辑”并不是单独章节，而是功能描述的一部分，应按业务复杂度决定详略：

- 简单交互：用 1 到 2 句写清触发、反馈和结果即可
- 中等复杂交互：补充交互顺序、关键状态变化和必要分支
- 高复杂交互：再细写触发条件、交互顺序、反馈方式、状态变化、关键分支或中断条件

保证文中涉及的页面、入口和区域名称能与当前 `Page Structure` 对得上。若 `Page Structure` 中没有，不得脑补，只能标记“待确认”或回退修正 `page`。

### 数据规范

如当前功能涉及关键数据规则，应明确：

- 数据来源
- 字段口径
- 展示格式
- 枚举或状态映射
- 脱敏 / 隐藏规则
- 同步、导入导出或一致性规则

如某个模块内存在复杂表单，相关字段口径、校验规则、默认值、联动关系、可编辑条件、提交限制等表单数据规范，应直接写在该模块的详细需求说明中，不单独拆成模块外的新章节。

如当前模块内存在复杂表单，还应补充表单相关的数据规范，至少覆盖：

- 字段定义与口径
- 必填 / 选填规则
- 默认值与回填规则
- 校验规则
- 枚举值或选项来源
- 提交、保存、草稿、回显或编辑时的数据处理规则

## 图表规则

- 图表统一使用 Mermaid
- 复杂流程、状态或多角色顺序交互必须补 Mermaid 图
- 需要补图时，直接将 Mermaid 代码块放进正文

## 分块生成

如统一路由入口或当前上下文判断需要分块：

- 先输出 PRD 分块计划
- 再逐块生成
- 最后汇总

规模过大时不得直接整篇硬写。

## 状态更新

- 更新 `latest_artifacts.prd`
- 如本轮 PRD 已成为稳定基线，刷新 `stable_baselines.prd`
- 如发现 `feature_list` 或 `page_structure` 稳定锚点滞后，先修正状态文件，再继续

## 完成定义

- 已严格按模板主骨架输出
- 详细需求说明已覆盖真实存在的关键展示、操作、输入、加载、弹窗、异常、交互逻辑和数据规则
- 文中出现的页面、入口和区域名称全部可回溯到 `Page Structure`
- 已继承 `Feature List` 的角色、范围与关键业务属性
- 复杂流程或状态已补 Mermaid 图
- 如规模过大，已先给出分块计划而不是直接整篇硬写

## 错误处理与回退

- 若稳定 Feature List 或 Page Structure 不足以支撑 PRD，回退到 `feat` 或 `page`
- 若工作项目存在多版本上游且稳定基线未更新到最新确认版，停止生成并先修复状态锚点
- 若文中出现 `Page Structure` 不存在的页面、区域、入口或模块，先删除猜测内容并回退修正事实源
- 若 PRD 与已确认范围发生重大冲突，停止交接并先修复事实上游

## 交接

- 将 PRD 交给 `rev` 或 `mock`
- 保持 PRD 的细节足以约束原型范围、关键流程和后续需求变更

## 方法论

### INVEST 原则

每个子功能描述应尽量满足：
- Independent：不依赖其他未完成功能就能理解
- Negotiable：描述目标和约束，不规定技术实现
- Valuable：能说清对用户/业务的价值
- Estimable：有足够细节让开发评估工作量
- Small：一个可独立交付的单元
- Testable：验收标准清晰

### Given/When/Then 验收标准

- Given（前置条件）：在什么状态下
- When（操作）：用户/系统做了什么
- Then（预期结果）：应该出现什么结果

### 异常场景覆盖清单

每个功能点至少考虑：权限异常、数据异常（必填为空）、状态异常、并发异常、网络异常、依赖异常、超时异常

### 非目标声明

本期不做的事项必须显式列出。  
格式：本 PRD 不覆盖 XXX，该需求计划在 vX.X 中评估。

### MECE 原则（场景覆盖）

对每个功能点的场景分析应尽量满足"相互独立、完全穷尽"：
- 列出所有正常路径和异常路径
- 检查是否存在遗漏的分支
- 检查是否存在重复描述的场景

## 前置检查

执行前先读取 `docs/project-status.json`。
1. 如果文件不存在，停止执行并告知用户："当前项目尚未初始化 ShitPM，请先输入'初始化 ShitPM'。"
2. 检查 `blockers` 是否为空，不为空则停止并告知用户。
3. 检查 `pending_confirmations` 是否为空，不为空则停止并告知用户。
4. 检查 `stable_baselines` 中对应的上游锚点文件是否存在，不存在则停止并告知用户。
5. 若未满足，不得继续推进。
