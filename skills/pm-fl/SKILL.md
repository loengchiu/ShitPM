# PM FL

## 目标

生成稳定功能清单，作为后续 `pm-ps`、`pm-prd`、`pm-rv` 的能力事实源。

功能清单不是研发任务拆解，也不是页面结构表。它负责回答：

- 这次版本到底提供哪些能力
- 这些能力分别属于哪些模块
- 哪些能力本期做，哪些后续做
- 哪些能力存在高风险来源或待确认点

## 输入

- 项目目录
- 当前项目简介或范围结论
- 思维导图中的范围、流程、角色、判断节点和风险点
- 必要的补充澄清信息
- 当前稳定锚点（如存在）

输入优先级：

`当前稳定锚点 > 当前项目简介 / 范围结论 > 思维导图 > 补充澄清信息`

若上游信息不足以稳定收敛能力边界，不得硬写功能清单。

## 输出

将功能清单写入工作项目的 `docs/feature-lists/` 目录。

结构参考：`templates/feature-list.md`

## 规则

- 功能清单主体统一采用页面 / 能力级颗粒度
- 不在这里提前拆成研发任务
- 不混用步骤级写法
- 主体应回答“系统提供哪些能力”，不负责展开“每一步怎么发生”
- 每个模块下补 `涉及角色`
- B 端场景下不必额外塞“端”字段；若角色差异重要，优先在模块级写 `涉及角色`
- 必须从思维导图中的业务主流程、角色参与关系、关键判断节点、状态分支和风险点中提炼能力边界
- 若某项能力暂时无法稳定映射到页面或页面能力，可在对应模块或功能点下标记 `待确认`
- 模块划分、功能点、范围和优先级必须足够清晰，能够稳定支撑后续 `pm-ps` 和 `pm-prd`
- 不把完整卡片集、访谈整表或历史材料整段塞回主表

## 主表字段

主表统一采用以下字段：

- `#`
- `功能点`
- `说明`
- `优先级`
- `版本`
- `PRD章节`

补充要求：

- 优先级统一使用中文：`高 / 中 / 低`
- `版本` 只写版本边界，不拿来重复优先级
- `PRD章节` 用于后续规格落位，不承担页面映射职责

## 颗粒度要求

- 只写“有什么能力”
- 不写“每一步怎么发生”
- 能力点应能稳定映射到页面或页面能力
- 能力点可以基于业务闭环推导页面承载方向，但不要求思维导图先给出页面结构
- 若当前还无法稳定映射页面承载，可标记 `待确认`，由 `pm-ps` 继续收敛

以下内容不得进入主表：

- 按钮点击
- 状态跳转过程
- 校验反馈
- 异常提示细节
- 页面区域树
- 原型页面映射
- 操作步骤

这些内容应下沉到 `pm-ps`、`pm-prd`、流程说明或原型。

## 推荐结构

- 模块结构
- 模块标题
- 涉及角色
- 功能清单主表
- 版本边界
- 关键需求摘要（可选）

说明：

- `页面 / 能力映射说明` 不再作为默认主结构
- 若页面归属是当前能力成立的关键前提，可以在 `说明` 中简短写“归属页面/页面能力”
- 但不得展开区域树，不得提前写成页面结构表

## 说明字段要求

- 只写能力边界和范围说明
- 不写页面区域细节
- 不写按钮级交互
- 不写操作步骤
- 不写原型表达方式
- 若某能力成立依赖明确页面归属，可简短写“归属页面/页面能力”，但不展开页面区域层级

## 版本边界要求

- 本期范围与后续规划必须分开表达
- 不把“优先级”和“版本边界”混成同一字段
- 若范围仍明显摇摆，不得交给 `pm-ps`

## 关键需求摘要

如需保留需求溯源，只在文末增加“关键需求摘要”：

- 只保留高风险、来源复杂、待确认较多的需求
- 不把完整卡片集重新塞回主文档
- 只保留对后续 `pm-prd` 和 `pm-rv` 真有约束价值的摘要

## 不负责

- 页面区域说明
- 原型页面映射
- 操作步骤细节
- 研发任务拆分

## 与思维导图衔接

- `pm-fl` 以上游思维导图中的业务闭环信息为输入
- 重点吸收：业务主流程、角色参与关系、关键判断节点、状态分支、风险点
- `pm-fl` 负责把业务闭环转成稳定能力清单
- 不要求思维导图先产出页面结构
- 若某项能力的页面承载关系暂时无法确定，应先标记 `待确认`，交由后续页面结构阶段继续收敛

## 状态更新

- 更新 `current_stage` 为 `pm-fl`
- 更新 `last_action`
- 更新 `next_recommended`，默认推荐 `pm-ps`
- 更新 `context_summary`
- 更新 `latest_artifacts.feature_list`
- 如本轮版本替代旧基线，刷新 `stable_baselines.feature_list`
- 如本轮需要用户拍板范围、优先级或模块边界，写入 `pending_confirmations`

## 完成定义

- 遵守 `contracts/done-criteria.md` 中 `pm-fl` 的完成标准
- 主表中的功能点必须保持页面 / 能力级颗粒度
- 不得混入步骤级条目
- 模块级必须写 `涉及角色`
- 主表字段完整
- 未混入页面区域或原型映射
- 版本边界明确
- 功能模块、功能点、优先级和版本边界足够稳定，才能进入 `pm-ps`
- 如果范围、模块结构或优先级仍明显摇摆，不得交给 `pm-ps`
- 未通过自检时，继续收敛或回退，不要硬写 PRD

## 错误处理与回退

- 遵守 `contracts/error-handling.md`
- 若上游简介、思维导图或补充澄清不足，回退到正确上游阶段
- 若功能点混入步骤级内容，要求重写
- 若用户拒绝确认明显存在分歧的范围边界，应说明这会直接影响 PRD 的事实源稳定性
- 若功能点无法映射到明确模块，应停止交接并先修复结构
- 若结构问题明显，不得继续进入 `pm-ps`

## 交接

- 将稳定功能清单交给 `pm-ps`
- 保持功能清单可被后续评审用于和 `PRD`、`Prototype` 做一致性校验
