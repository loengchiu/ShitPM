# PM RV

## 目标

检查 `Feature List`、`Page Structure` 和 `PRD` 三者是否一致，判断是否需要：

- 回写上游
- 回退阶段
- 修正稳定锚点

支持两种评审模式，根据用户意图自动判定，无需额外指令。

## 模式判定

在执行任何评审前，先判定当前模式：

- **自检模式**（默认）：用户未提及具体角色或开发侧，AI 独立完成一致性检查
- **多角色评审模式**：用户提及角色（如"拉业务方"、"和前后端对齐"、"开评审会"），切换到多角色评审流程

---

## 自检模式

### 输入优先级

1. 用户显式给出的当前文件路径
2. `stable_baselines`
3. 目录扫描结果

### 评审范围

- `pm-fl`
- `pm-ps`
- `pm-prd`

### 必做检查

#### 1. 版本关系

- 当前显式路径与稳定锚点是否一致
- 是否存在新版已生成但稳定锚点仍指向旧版

#### 2. 产物一致性

- Feature List 是否仍然代表当前能力边界
- Page Structure 是否仍然代表当前页面承载边界
- PRD 是否越权重新定义能力或页面结构

#### 3. 回写判定

若 PRD 人工改稿后：

- 只影响规格表达：停留在 PRD
- 影响页面承载：回写 `pm-ps`
- 影响能力边界：回写 `pm-fl`

### 输出要求

必须输出：

- 实际采用的当前路径
- 与 `stable_baselines` 的关系
- 当前问题属于哪一层
- 是否需要回写
- 是否需要回退
- 推荐下一步动作

## 多角色评审模式

### 角色与默认视角

| 角色 | 关注域 | 上下文不足时 |
|------|--------|--------------|
| 业务经理 | 业务目标、收益假设、优先级 | 降级为业务假设确认，不强求精确 ROI |
| 项目经理 | 排期依赖、风险、跨团队接口 | 先识别关键依赖与 owner |
| 产品经理 | 需求闭环、边界、版本范围 | 优先补齐主流程与异常流 |
| 前端 | 交互复杂度、组件复用、联动逻辑 | 按状态流和组件边界评估 |
| 后端 | 接口面、数据模型、权限 | 先抽象实体、动作、权限 |
| UI | 设计一致性、视觉基线、可实现性 | 先检视信息层级与状态覆盖 |
| 测试 | 关键路径、可测试性、边界情况 | 先识别不可测点 |

每个角色输出固定三层：关注域、必问问题、结论。结论只能是 `通过项 / 待处理项 / 待确认假设`。

### 角色问题模板

| 角色 | 必问问题 | 通过标准 |
|------|----------|----------|
| 业务经理 | 是否服务本期核心目标；优先级边界是否清晰；是否至少给出收益方向或业务假设 | 目标明确，边界清楚 |
| 项目经理 | 是否存在关键依赖；高风险是否有 owner；是否冲击时间线 | 依赖可追踪，风险已暴露 |
| 产品经理 | 主异常流程是否完整；是否有越界内容；是否仍与上游锚点一致 | 需求闭环，无关键歧义 |
| 前端 | 状态与联动是否清晰；组件是否可复用；高复杂交互是否有约束 | 交互可实现，状态可收敛 |
| 后端 | 实体关系是否足够；接口动作面是否完整；权限审计状态流是否遗漏 | 数据与接口支撑主流程 |
| UI | 信息层级是否一致；是否存在不可实现设计；关键状态是否覆盖 | 设计一致，风险可控 |
| 测试 | 关键路径是否可测；边界异常权限是否可验证；是否存在难造数难观测问题 | 关键路径可测，无明显盲区 |

### B 端特例

- 业务经理默认不回答精确 ROI、市场规模或经营指标。
- 只强制确认两件事：本期核心业务目标、本期明确不做什么。
- 无法回答的业务信息记为 `待业务确认假设`，不直接判阻塞。
- 仅当未确认信息影响版本范围、优先级或立项时，才升级为阻塞。

### 角色选择规则

#### 默认组合

| 评审对象 | 默认角色 | 补充说明 |
|----------|----------|----------|
| Feature List | 产品经理、业务经理、项目经理 | 若涉及复杂权限或数据流，再加后端 |
| Page Structure | 产品经理、前端、UI | 若跨页面状态联动复杂，再加后端 |
| PRD | 产品经理、前端、后端、测试 | 若涉及上线窗口或多团队协作，再加项目经理 |
| HTML 原型 / 标注稿 | 产品经理、前端、UI、测试 | 若隐含接口行为或权限逻辑，再加后端 |

对于大多数 B 端项目，用户只说“开评审”时，优先推荐：产品经理、前端、后端、测试。

#### 增减条件

| 角色 | 追加条件 | 可省略条件 |
|------|----------|------------|
| 业务经理 | 存在业务策略、计费、优先级冲突，争议是“值不值得做” | 仅是内部提效、后台配置、体验修订 |
| 项目经理 | 存在上线窗口、外部承诺、跨团队依赖 | 仍在早期澄清，未进入排期承诺 |
| 前端 | 有复杂交互、批量操作、页面联动、前端校验 | 尚未进入页面承载设计 |
| 后端 | 有新实体、新接口、状态机、权限、跨系统约束 | 仅做信息组织或视觉走查 |
| UI | 评审对象是页面结构、原型、高保真；需确认视觉承载 | 仅有文字 PRD |
| 测试 | 已进入 PRD/原型；存在复杂权限、异常流、高回归风险 | 仅讨论上游能力边界 |

快速评审或信息不足时，缩减顺序固定为：先去业务经理和项目经理，再按产物保留最小闭环；产品经理必须保留。最小闭环建议：Feature List 为“产品经理 + 业务假设确认方”，Page Structure 为“产品经理 + 前端”，PRD 为“产品经理 + 前端 + 后端”，原型/标注稿为“产品经理 + 前端 + UI”。

自动推荐角色时，必须同时说明：当前评审对象、默认组合、本次追加角色及原因、本次省略角色及原因、是否启用 B 端降级策略。

### 结论判定

- `通过`：与上游锚点一致，无明显越权扩展；关键问题已有答案；无高风险未决项。
- `带条件通过`：主体方案成立，但仍有少量待修订项或待确认假设；这些问题不阻断当前推进。输出时必须列出条件项、责任人、最晚补齐时点。
- `阻塞`：范围或承载边界存在根本冲突；关键 Tiger 问题未解；缺失信息已影响实现、优先级或立项判断；或当前产物与稳定锚点明显失配。输出时必须列出阻塞问题、归属层级 `pm-fl / pm-ps / pm-prd / 待用户回答`、推荐路由 `pm-fix / pm-scope / 当场修订 / 问答模式`。

### 大产物评审的上下文控制

- 先缩范围，再读内容：先确认评审对象、评审目标、是否只评某一部分；若用户限定范围，不外扩。
- 只取最小片段：优先读取目录结构、目标章节、相关段落、版本与锚点信息，不一次性加载整份文档。
- 超长内容分块：先全局扫描风险热点，再按章节/页面组/能力组逐块评审，最后汇总跨块结论。
- 输出以问题为主：少引原文，不贴长摘录；同类问题复用编号，默认顺序是结论、问题、待确认假设、必要时的短摘要。
- 上下文不足时主动降级：只做范围判定与热点扫描，只评指定章节，或只评部分角色，并明确未纳入本轮评审的范围。

若用户未明确列举角色，根据当前阶段产物推荐默认角色组合，并请用户确认后再执行。

### 执行流程

1. **确认范围与角色**：确定评审对象、版本、章节范围与参与角色。
2. **小范围扫描**：先读最小必要片段，必要时分块。
3. **逐角色输出**：按角色给出通过项、待处理项、待确认假设。
4. **判定结论与路由**：输出结论、问题归属、是否回写/回退。

### 输出格式

```
## 评审结论：[通过 / 带条件通过 / 阻塞]

### [角色名] 视角
- ✅ 通过项
- ⚠️ 待处理项
- ？待确认假设

...

### 待处理问题汇总
1. [问题描述] — 路由：[需要回答 / pm-fix / pm-scope / 当场修订]
```

写入 `docs/review-meetings/YYYY-MM-DD_review-<topic>.md`

## 评审后问答模式

若评审结束后存在**需要用户回答**的待处理问题，自动进入问答模式：

- 不将问题列表一次性丢给用户
- 逐条提问，等待用户回答后再继续下一条
- 所有回答收集完毕后，汇总写入评审记录

### 问题定性（每条待处理问题必须先定性）

在逐条提问前，先请用户对每条问题做定性判断：

| 定性 | 含义 | 处理方式 |
|------|------|---------|
| **Tiger**（真实风险）| 确认存在、必须解决，否则阻塞推进 | 写入 `blockers`，路由到对应 Skill 处理 |
| **Paper Tiger**（过度担忧）| 表面是问题，实际影响可忽略或已有兜底 | 记录但不阻塞，附说明写入评审记录 |
| **Elephant**（待观察）| 不确定是否真实问题，团队尚未充分讨论 | 写入 `pending_confirmations`，标注待跟进 |

若用户无法判定，默认标记为 **Tiger**，优先处理。

### 路由规则（定性完成后执行）

- Tiger 且回答解决了问题 → 当场修订，标记为已解决
- Tiger 且涉及锚点变动 → 路由到 `pm-fix`
- Tiger 且涉及范围边界 → 路由到 `pm-scope`
- Tiger 且暂无答案 → 写入 `blockers`
- Paper Tiger → 记录说明，不阻塞，继续下一条
- Elephant → 写入 `pending_confirmations`，等待后续跟进
