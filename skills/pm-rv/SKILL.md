# PM RV

## 目标

检查 `Feature List`、`Page Structure` 和 `PRD` 三者是否一致，判断是否需要：

- 回写上游
- 回退阶段
- 修正稳定锚点

支持两种评审模式，根据用户意图自动判定，无需额外指令。

## 模式判定

在执行任何评审前，先判定当前模式：

- **自检模式**（默认）：用户未提及具体角色或开发侧，AI 独立完成一致性检查
- **多角色评审模式**：用户提及角色（如"拉业务方"、"和前后端对齐"、"开评审会"），切换到多角色评审流程

---

## 自检模式

### 输入优先级

1. 用户显式给出的当前文件路径
2. `stable_baselines`
3. 目录扫描结果

### 评审范围

- `pm-fl`
- `pm-ps`
- `pm-prd`

### 必做检查

#### 1. 版本关系

- 当前显式路径与稳定锚点是否一致
- 是否存在新版已生成但稳定锚点仍指向旧版

#### 2. 产物一致性

- Feature List 是否仍然代表当前能力边界
- Page Structure 是否仍然代表当前页面承载边界
- PRD 是否越权重新定义能力或页面结构

#### 3. 回写判定

若 PRD 人工改稿后：

- 只影响规格表达：停留在 PRD
- 影响页面承载：回写 `pm-ps`
- 影响能力边界：回写 `pm-fl`

### 输出要求

必须输出：

- 实际采用的当前路径
- 与 `stable_baselines` 的关系
- 当前问题属于哪一层
- 是否需要回写
- 是否需要回退
- 推荐下一步动作

## 多角色评审模式

### 支持角色

| 角色 | 评审焦点 |
|------|---------|
| 业务经理 | 业务目标覆盖、ROI、优先级边界 |
| 项目经理 | 时间线依赖、风险、跨团队接口 |
| 产品经理 | 需求完整性、边界情况、版本范围 |
| 前端 | 交互复杂度、组件复用性、联动逻辑 |
| 后端 | API 接口面、数据模型、权限设计 |
| UI | 设计一致性、视觉基线对齐、可实现性 |
| 测试 | 关键路径覆盖、可测试性、边界情况 |

若用户未明确列举角色，根据当前阶段产物推荐默认角色组合，并请用户确认后再执行。

### 执行流程

1. **确认评审范围**：当前阶段产物 + 参与角色
2. **逐角色生成视角**：为每个角色生成专属问题清单
3. **逐角色呈现问题**：按角色分组输出，不混在一起
4. **输出评审结论**：通过 / 带条件通过 / 阻塞，并列出所有待处理问题

### 输出格式

```
## 评审结论：[通过 / 带条件通过 / 阻塞]

### [角色名] 视角
- ✅ 通过项
- ⚠️ 待处理项

...

### 待处理问题汇总
1. [问题描述] — 路由：[需要回答 / pm-fix / pm-scope / 当场修订]
```

写入 `docs/review-meetings/YYYY-MM-DD_review-<topic>.md`

## 评审后问答模式

若评审结束后存在**需要用户回答**的待处理问题，自动进入问答模式：

- 不将问题列表一次性丢给用户
- 逐条提问，等待用户回答后再继续下一条
- 所有回答收集完毕后，汇总写入评审记录

### 问题定性（每条待处理问题必须先定性）

在逐条提问前，先请用户对每条问题做定性判断：

| 定性 | 含义 | 处理方式 |
|------|------|---------|
| **Tiger**（真实风险）| 确认存在、必须解决，否则阻塞推进 | 写入 `blockers`，路由到对应 Skill 处理 |
| **Paper Tiger**（过度担忧）| 表面是问题，实际影响可忽略或已有兜底 | 记录但不阻塞，附说明写入评审记录 |
| **Elephant**（待观察）| 不确定是否真实问题，团队尚未充分讨论 | 写入 `pending_confirmations`，标注待跟进 |

若用户无法判定，默认标记为 **Tiger**，优先处理。

### 路由规则（定性完成后执行）

- Tiger 且回答解决了问题 → 当场修订，标记为已解决
- Tiger 且涉及锚点变动 → 路由到 `pm-fix`
- Tiger 且涉及范围边界 → 路由到 `pm-scope`
- Tiger 且暂无答案 → 写入 `blockers`
- Paper Tiger → 记录说明，不阻塞，继续下一条
- Elephant → 写入 `pending_confirmations`，等待后续跟进
