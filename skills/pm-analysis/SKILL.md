# PM Analysis

## 作用

使用这个技能把范围澄清后的材料整理成正式分析产物。这个阶段的目标不是直接写 PRD，而是先沉淀稳定的项目简报，明确项目总览、资料分析结论、关键流程文字版、关键规则、二次访谈补充与修正、变更与修正留痕、风险与待确认项，供后续思维导图、功能清单和 PRD 复用。

使用以下参考文件：

- `../../contracts/artifact-schema.md`
- `../../contracts/done-criteria.md`
- `../../contracts/error-handling.md`
- `../../templates/project-brief-lite.md`

## 输入

按以下优先级读取：

1. 当前项目目录与已有上下文
2. 范围澄清结论
3. 来源材料索引
4. 会议纪要
5. 原始需求点、补充说明与参考材料
6. 二次访谈补充信息
7. 当前稳定锚点与最新产物

## 输出

将分析阶段产物写入工作项目的 `docs/briefs/` 目录。

## 规则

- 先形成正式项目简报，再进入思维导图。
- 项目简报必须按模板显式包含“资料分析结论”章节，区分已确认事实、高可信推断和存在冲突或待核实的信息。
- 项目简报必须按模板显式包含“二次访谈补充与修正”章节，记录新增信息、对原理解的修正和仍待确认的问题。
- 项目简报必须按模板显式包含“变更与修正留痕”章节，统一记录需求变更、理解修正和访谈补充。
- 项目简报聚焦“项目总览 + 资料分析结论 + 关键流程文字版 + 变更与修正留痕”，不要把它扩写成第二份 PRD。
- 价值说明保持轻量表达，不强行扩成复杂价值模型。
- 关键流程采用文字描述，不要求在本阶段补流程图。
- 待确认问题必须显式保留，不要在信息不足时擅自补全。
- 如分析发现问题定义本身不成立、边界互相冲突，必须明确标记为风险点，而不是带着问题继续进入下游。
- 分析产物必须足以支撑后续思维导图；如果还不能支撑，应停止推进并说明缺口。

## 不负责

- 不直接生成 PRD
- 不输出页面结构
- 不设计原型
- 不替代思维导图阶段

## 状态更新

- 进入本阶段时，将当前阶段设为 `pm-analysis`
- 更新 `last_action`
- 更新 `next_recommended` 为 `pm-mm`
- 更新 `context_summary`
- 将本轮项目简报写入 `latest_artifacts.briefs`
- 将仍影响推进的待确认项写入 `pending_confirmations`

## 完成定义

- 已形成正式项目简报
- 简报包含背景、资料分析结论、范围、角色、关键流程、关键规则、二次访谈补充与修正、变更与修正留痕、风险与待确认项
- 已区分已确认事实、高可信推断和冲突/待核实信息
- 未扩写成第二份 PRD
- 产物足以支撑后续思维导图
- 若待确认项足以阻塞下游，未将本阶段标记为完成

## 错误处理与回退

- 若范围澄清结果不足以支撑分析，回退到 `pm-scope`
- 若来源材料不足以整理关键流程、关键规则或角色边界，停止推进并要求补充材料
- 若二次访谈补充导致原有理解发生明显冲突，先修订项目简报，不直接进入思维导图
- 若分析过程中发现核心问题定义冲突，明确记录风险并停止继续下游

## 交接

- 将项目简报交给 `pm-mm`
- 如分析结果仍无法支撑思维导图，先继续补充分析，不直接交接
