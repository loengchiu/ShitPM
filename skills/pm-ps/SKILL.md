# PM PS

## 目标

基于功能清单收敛页面与区域结构，作为 PRD 与 Prototype 的共同页面事实源。

## 输入

- 稳定功能清单
- 思维导图中的业务闭环信息（用于校验页面承载是否偏离业务主线）
- 当前稳定锚点（如存在）

## 规则

- 上游默认优先认当前稳定功能清单，并用思维导图中的业务闭环信息校验页面划分是否合理。
- `Page Structure` 是 PRD 与 Prototype 的共同页面事实源，后续两个阶段都应围绕它展开，而不是各自发明页面体系。
- 如果页面结构设计与业务闭环明显冲突，先标记 `待确认`，不要私自覆盖。
- `pm-ps` 只负责能力到页面与区域的承载映射，不负责修现有 PRD 或 Prototype；如发现下游产物与页面结构冲突，交给 `pm-rv` / `pm-fix` 处理。
- 每个页面只写：
  - 页面目标
  - 页面结构树
- 每个区域固定描述：
  - 对应功能点
  - 作用
  - 关键操作
  - 关键状态

## 页面目标要求

- 每个页面必须有明确页面目标
- 页面目标只说明该页面承载什么，不解释业务背景
- 页面名称和页面目标应能回溯到功能清单中的能力分组，并与思维导图中的业务闭环保持一致；若无法回溯，标记 `待确认`

## 区域描述要求

- `对应功能点` 只写编号或名称，不重复抄功能清单说明
- `作用` 只写该区域承担的职责
- `关键操作` 只写区域级操作
- `关键状态` 只写会影响页面理解或后续需求文档的状态

## 边界

- 页面结构只回答“这些能力放在哪、由哪些区域承载”
- 不回答优先级、版本边界和能力价值判断
- 不承担交互细节正文解释
- 不脱离思维导图和功能清单自行新增大段页面体系

## 禁止

- 页面总表
- 结构稳定性结论
- 使用说明
- 自检文字

## 与上游衔接

- 页面结构阶段以上游功能清单为直接输入，不要求思维导图先给出页面结构
- 页面结构阶段负责根据能力分组和业务闭环，把能力放到合理页面与区域中
- 若页面划分无法从功能清单和业务闭环稳定推出，应停在 `待确认`

## 与下游衔接

- `pm-prd` 应以 `Page Structure` 作为页面承载事实源，不再重新发明页面结构
- `pm-pt` 应以 `Page Structure` 作为页面组织和关键区域的共同锚点，不得脱离其新增主页面或关键区域

## 状态更新

- 更新 `latest_artifacts.page_structure`
- 如本轮版本替代旧基线，刷新 `stable_baselines.page_structure`
