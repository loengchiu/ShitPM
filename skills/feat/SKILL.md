# PM FL

## 目标

生成稳定功能清单，回答这次版本到底提供哪些能力、这些能力属于哪些模块、哪些本期做、哪些后续做。功能清单不是研发任务拆解，也不是页面结构表。

## 输入

- 项目目录
- 当前项目简报或范围结论
- 思维导图中的范围、流程、角色、判断节点和风险点
- 必要的补充澄清信息
- 当前稳定锚点（如存在）

输入优先级：

`当前稳定锚点 > 当前项目简报 / 范围结论 > 思维导图 > 补充澄清信息`

若上游信息不足以稳定收敛能力边界，不得硬写功能清单。

## 输出

将功能清单写入工作项目的 `docs/feature-lists/` 目录。  
结构参考：`../../templates/feature-list.md`

## 规则

- 主体统一使用页面/能力级颗粒度。
- 不拆成研发任务，不混入步骤级内容。
- 每个模块写清 `涉及角色`。
- 必须从思维导图中的业务主流程、角色参与、关键判断节点、状态分支和风险点中提炼能力边界。
- `说明` 字段只写能力边界、范围与限制。
- 如果页面归属是该能力成立的前提，可以在 `说明` 中简短写“归属页面/页面能力”；除此之外不展开页面结构。
- 能力尚不能稳定映射到页面承载时，标记 `待确认`，留给 `page` 收敛。
- 模块划分、功能点、优先级和版本边界必须足够清楚，能够稳定支撑 `page` 和 `prd`。
- 每个功能点都应保留“能力字段线索”，说明这项能力离不开哪些关键字段；这里只保留线索，不做正式定义。
- 不把完整卡片集、访谈整表或历史材料整段塞回主表。

## 主表字段

- `#` | `功能点` | `说明` | `优先级`（高 / 中 / 低）
- `版本`（只写版本边界，不重复优先级）
- `PRD章节`（用于后续规格落位，不承载页面映射）
- `归属页面`（必填，写承载候选页面或承载方式） | `写作提示` | `写作状态`

## 颗粒度要求

- 只回答“有什么能力”，不回答“每一步怎么发生”
- 不提前写页面区域、按钮交互、原型映射或操作步骤
- 主表不得写入：按钮点击、状态跳转、校验反馈、异常提示细节、页面区域树、原型页面映射、操作步骤

## 推荐结构

- 模块结构 + 模块标题 + 涉及角色
- 功能清单主表 + 版本边界 + 关键需求摘要（可选）

## 关键需求摘要

仅在文末保留少量高风险、来源复杂或待确认较多的需求摘要：

- 只保留对后续 `prd` 和 `rev` 真有约束价值的内容
- 不把完整卡片集重新堆回正文；仅保留高风险或高不确定项

## 不负责

- 页面区域说明、原型页面映射、操作步骤细节、研发任务拆分

## 与思维导图衔接

- `feat` 以上游思维导图中的业务闭环信息为输入
- 重点吸收：业务主流程、角色参与关系、关键判断节点、状态分支、风险点
- `feat` 负责把业务闭环转成稳定能力清单
- 若某项能力的页面承载暂时无法确定，应标记 `待确认`，交由 `page` 继续收敛

#### 归属页面

- 每个功能点必须标注承载候选页面或承载方式，但这里不要求正式 `PS-__` 编号
- 优先写自然语言页面名称；未稳定时标注 `待确认`
- 最小示例：`供应商列表页` / `同时涉及：详情页` / `横切规则`
- 正式页面编号和区域映射由 `page` 阶段收敛，`feat` 不得依赖 `Page Structure` 反向定义自己

#### 写作提示

- 每个功能点必须包含写作提示；写作提示必须是可执行提示，不是元指令或空话
- 精简格式：关联字段线索 / 关键交互（不超过 3 步） / 关键校验 / 关键权限 / 关键异常
- `关联字段线索` 只写字段名称或口径关键词，不写正式 `DD-__` 编号，不要求提前统一命名
- 最小门禁：禁止使用“需要描述”“需要定义”等元指令；禁止写完整交互流程、技术实现方案、页面区域细节

#### 跨模块功能点

- 不属于单个模块的功能点归入"跨模块功能点"章节
- 跨模块功能点标注涉及的所有页面
- 在 PRD 中写入"横切规则"章节，不写入某个页面章节

#### 写作状态

- 每个功能点包含写作状态字段：☐ 未写
- 写作状态供 PM 人工编写 PRD 时追踪进度
- AI 生成 PRD 时忽略此字段

## 完成自审

功能清单生成完毕后，逐条检查以下项目，不通过的必须修正：

1. `能力边界`：每个功能点是否写清了做什么和不做什么？是否存在只有动词没有宾语的模糊表述？
2. `上游追溯`：每个功能点是否能在思维导图中找到对应的业务节点？是否存在思维导图中没有对应来源的功能点？
3. `角色覆盖`：每个功能点是否标注了涉及角色？是否存在没有角色的功能点？
4. `版本边界`：本期/后续版本划分是否清晰？是否存在待定或看情况的模糊项？
5. `跨模块检查`：跨模块功能点是否已识别并归入“跨模块功能点”章节？
6. `重复检查`：是否存在两个功能点描述的是同一件事，不同措辞但同一能力？
7. `写作提示检查`：每个功能点的写作提示是否包含具体内容？是否存在需要描述、需要定义等元指令？
8. `字段线索检查`：每个关键能力是否保留了必要的字段线索？是否存在访谈或思维导图中明显提到的重要字段，在功能点中完全消失？

执行规则：

- 自审发现的问题先直接修正
- 若问题仅是表述不清，留在当前稿件内修正
- 若问题说明上游思维导图本身不稳定或缺失事实，停止推进并回退到 `mind`
- 全部通过后，才允许进入状态更新

## 状态更新

- 更新 `current_stage` 为 `feat`
- 更新 `last_action`
- 更新 `next_recommended`，默认推荐 `page`
- 更新 `context_summary`
- 更新 `latest_artifacts.feature_list`
- 如本轮版本替代旧基线，刷新 `stable_baselines.feature_list`
- 如本轮需要用户拍板范围、优先级或模块边界，写入 `pending_confirmations`

## 完成定义

- 遵守 `contracts/done-criteria.md` 中 `feat` 的完成标准
- 主表功能点保持页面/能力级颗粒度
- 模块级已写 `涉及角色`
- 主表字段完整
- 未混入页面区域、原型映射或步骤级内容
- 版本边界明确
- 功能模块、功能点、优先级和版本边界足够稳定，能够进入 `page`

## 错误处理与回退

- 遵守 `contracts/error-handling.md`
- 若上游简报、思维导图或补充澄清不足，回退到正确上游阶段
- 若功能点混入步骤级内容，要求重写
- 若用户拒绝确认明显存在分歧的范围边界，应说明这会直接影响 PRD 的事实源稳定性
- 若功能点无法映射到明确模块，应停止交接并先修复结构

## 交接

- 将稳定功能清单交给 `page`
- 保持功能清单可被后续评审用于和 `PRD`、`Prototype` 做一致性校验

## 方法论：优先级评估与能力边界

### 简化 RICE 评分

仅当需要排序时启用：
- 各维度 1-3 分：R 覆盖面 / I 影响度 / C 信心 / E 成本（反向）
- 综合评分 = (R × I × C) / E
- ≥ 6：高；3-5：中；≤ 2：低或考虑砍掉
- 仅作快速排序辅助；分数相近时优先覆盖面大、信心高的功能点

### 版本边界三问

对标记为"低"或"中"的功能点，逐个回答：
- 砍掉它，核心业务闭环还能跑通吗？→ 能，就坚决移到后续版本
- 它是某个高优功能的前置依赖吗？→ 不是，就可以延后
- 它是"有了更好"还是"没有不行"？→ 前者延后，后者保留

### 能力 vs 实现分离

- 功能清单只写"系统具备什么能力"，不写"怎么实现"
- "供应商可以提交资质材料"是能力 → 写入功能清单
- "供应商填写表单后点提交按钮，系统校验必填项"是实现 → 不写入功能清单
