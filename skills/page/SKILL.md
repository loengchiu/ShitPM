# PM PS

## 目标

基于功能清单收敛页面与区域结构，作为 PRD 与 Prototype 的共同页面事实源。

## 输入

- 稳定功能清单
- 思维导图中的业务闭环信息（用于校验页面承载是否偏离业务主线）
- 当前稳定锚点（如存在）

## 规则

- 上游默认优先认当前稳定功能清单，并用思维导图中的业务闭环信息校验页面划分是否合理。
- `Page Structure` 是 PRD 与 Prototype 的共同页面事实源，后续两个阶段都应围绕它展开，而不是各自发明页面体系。
- 如果页面结构设计与业务闭环明显冲突，先标记 `待确认`，不要私自覆盖。
- `page` 只负责能力到页面与区域的承载映射，不负责修现有 PRD 或 Prototype；如发现下游产物与页面结构冲突，交给 `rev` / `fix` 处理。
- 每个页面只写：
  - 页面目标
  - 页面结构树
- 每个区域固定描述：
  - 对应功能点
  - 作用
  - 包含元素清单
  - 关键操作
  - 关键状态

#### 页面目标

- 每个页面必须包含一句话目标：用户打开这个页面是为了完成什么事
- 页面目标不是功能描述，而是用户视角的使用目的
- 示例："采购员查看和管理所有已提交的准入申请"

#### 包含元素清单

- 每个区域必须列出所有包含的 UI 元素
- 元素清单只做“盘点”，只列出有什么，不写完整行为句子
- 格式：[元素类型] [元素名称]
- 元素类型限定为：按钮、输入框、下拉框、表格、标签、标签页、开关、复选框、单选框、文件上传、日期选择器、搜索框、链接、文本展示、图标、分隔线、弹窗
- 不允许出现"其他"、"等"模糊表述

#### 页面间关系

- 必须显式标注所有页面间的跳转关系（从哪个元素跳到哪个页面，传什么参数）
- 必须显式标注跨页面的数据依赖
- 必须标注多页面共享数据的主数据源
- 页面间关系写入独立的"页面间关系"章节，不散落在各页面描述中

#### 写作提示

- 每个区域包含写作提示，格式与 Feature List 一致：
  - 关联字段（字段名称或字段口径关键词）、关键交互、关键校验、关键异常
- `关联字段` 是下游 `prd` 收敛数据字典的直接线索；每个涉及表单、表格、详情展示或筛选条件的区域都必须填写
- 写作提示只列关键项，不列常规项
- 页面级注意事项单独列出（跨区域交互、页面级异常、页面级权限）

#### 写作状态

- 每个区域包含写作状态字段：☐ 未写
- 供 PM 人工编写 PRD 时追踪进度
- AI 生成时忽略

#### PRD 章节映射

- 每个页面标注 PRD 对应章节编号（如 5.1、5.2）
- 页面顺序即为 PRD 第 5 章的章节顺序
- 页面总览中列出所有页面的章节映射关系

## 结构表达要求

- 页面名称和页面目标应能回溯到功能清单中的能力分组，并与思维导图中的业务闭环保持一致；若无法回溯，标记 `待确认`
- `对应功能点` 只写编号或名称，不重复抄功能清单说明
- `作用` 只写该区域承担的职责
- `包含元素清单` 只列元素，不写行为说明
- `关键操作` 只写区域级主操作，控制在 1-3 个
- `关键状态` 只写会影响页面理解或后续需求文档的状态，控制在 1-3 个

## 边界

- 页面结构只回答“这些能力放在哪、由哪些区域承载”
- 不回答优先级、版本边界和能力价值判断
- 不承担交互细节正文解释
- 不脱离思维导图和功能清单自行新增大段页面体系

## 禁止

- 页面总表
- 结构稳定性结论
- 使用说明
- 自检文字

## 与上游衔接

- 页面结构阶段以上游功能清单为直接输入，不要求思维导图先给出页面结构
- 页面结构阶段负责根据能力分组和业务闭环，把能力放到合理页面与区域中
- 若页面划分无法从功能清单和业务闭环稳定推出，应停在 `待确认`
- 每个区域必须关联至少一个功能点（来自 Feature List）
- Feature List 中的每条功能点必须能在 Page Structure 中找到归属区域
- 如果两边对不上，说明有一边需要修正

## 与下游衔接

- `prd` 应以 `Page Structure` 作为页面承载事实源，不再重新发明页面结构
- `mock` 应以 `Page Structure` 作为页面组织和关键区域的共同锚点，不得脱离其新增主页面或关键区域

## 完成自审

页面结构生成完毕后，必须执行以下自审；不通过不得进入状态更新：

1. `双向覆盖`：Feature List 中每个功能点是否都能在 Page Structure 中找到承载页面和区域？Page Structure 中是否存在没有对应功能点的页面或区域？
2. `页面目标`：每个页面是否都有明确的一句话目标，且是用户视角而不是功能视角？
3. `元素完整性`：每个区域的元素清单是否完整？是否存在“其他”“等”模糊表述？非自解释元素是否有一句话说明其业务含义？
4. `页面关系`：页面跳转关系、跨页面数据流、主数据源是否已显式标注？
5. `区域归属`：每个区域是否都关联了至少一个功能点？是否存在“不知道放哪”的区域？
6. `PRD 映射`：每个页面是否标注了 PRD 章节编号？页面顺序是否可直接支撑 PRD 第 5 章？
7. `写作提示检查`：每个区域的写作提示是否包含具体内容？是否存在需要描述、需要定义等元指令？
8. `字段线索检查`：所有涉及表单、表格、详情展示或筛选条件的区域，是否都写了 `关联字段`？是否存在区域有数据承载但没有字段线索？

执行规则：

- 自审发现的问题先直接修正
- 若问题仅是页面表达不清，留在当前稿件内修正
- 若问题说明 Feature List 本身存在缺口、重复或能力边界错误，停止推进并回退到 `feat`
- 全部通过后，才允许进入状态更新

## 状态更新

- 更新 `latest_artifacts.page_structure`
- 如本轮版本替代旧基线，刷新 `stable_baselines.page_structure`

## 方法论：信息架构与页面承载

### 页面划分原则

- 一个页面 = 一个用户目标（用户打开这个页面是为了完成什么事）
- 如果一个页面同时服务两个以上完全不同的用户目标，考虑拆分
- 如果用户需要频繁在两个页面间切换才能完成一件事，考虑合并

### 区域组织原则

- 页面内区域按"操作频率"排列：高频操作 > 常规信息 > 低频设置
- 信息展示区域和操作区域物理分离，不要混在一起
- 每个区域必须有明确的功能点归属，不允许存在"不知道放哪就先放这"的区域

### 页面间关系原则

- 明确页面之间的导航关系（平级 / 上下级 / 弹出层）
- 跨页面的数据流转必须显式标注（从哪个页面带什么参数到哪个页面）
- 同一数据在多个页面出现时，明确哪个页面是"主数据源"，其他页面是"只读引用"

### 承载一致性校验

- 功能清单中的每个功能点必须能找到对应的页面和区域
- 如果某个功能点找不到承载位置，要么新增区域，要么回退修正功能清单
- 页面结构中不应存在功能清单中没有对应功能点的区域

## 前置检查

执行前先读取 `docs/project-status.json`。
1. 如果文件不存在，停止执行并告知用户："当前项目尚未初始化 ShitPM，请先输入'初始化 ShitPM'。"
2. 检查 `blockers` 是否为空，不为空则停止并告知用户。
3. 检查 `pending_confirmations` 是否为空，不为空则停止并告知用户。
4. 若未满足，不得继续推进。
