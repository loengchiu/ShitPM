# PM Scope

## 目标

在 PM 主链开始前，先完成最小必要的范围探索和拆分判定，避免直接进入后续产物生成。

## Hard Gate

在 `pm-scope` 完成并得到用户确认前：

- 不进入 `pm-mm` / `pm-fl` / `pm-ps` / `pm-prd`
- 不产出 spec、Feature List、Page Structure 或 PRD
- 不把尚未确认的拆分结论当作既定范围

`pm-scope` 的终点不是“AI 自己判断完”，而是“范围结论已向用户呈现并得到确认”，随后再回流 `pm-go`。

## 执行顺序

每次执行必须按以下顺序推进：

1. 先读取现有上下文：用户输入、已有产物、项目状态、相关文档
2. 先做过大范围识别，判断是否需要拆子项目
3. 若信息不足，逐轮追问，一次只问一个问题
4. 形成范围结论或拆分方案，给出推荐项和理由
5. 等用户确认后，再回流 `pm-go`

若在第 2 步就已确认任务明显过大，不要继续追问细节，应先完成拆分。

## 负责内容

- 澄清当前任务目标
- 判断是否属于单一 PM 任务
- 判断是否需要拆成多个子项目
- 形成轻量项目简介所需的基础信息

## 交互原则

- 先看上下文，再提问，不凭首句直接进入产物生成
- 一次只问一个问题，避免一轮抛出整套问卷
- 优先问会影响范围边界、拆分判断、优先级排序的问题
- 能用选项题时优先用选项题
- 若已有足够信息得出范围结论，就停止追问，转为确认

## 拆分优先原则

若需求同时覆盖多个相对独立的业务域、流程闭环或交付模块，优先做拆分，不在一个大任务里继续细化所有子域。

拆分时至少给出：

- 拆分方案
- 每个子项目的目标与边界
- 依赖关系与推荐顺序
- 当前建议优先推进的子项目

如存在多种合理拆分方式，应给出 2-3 种方案，说明取舍并给出推荐。

## 输出要求

每次执行后至少明确：

- 当前任务目标
- 范围边界
- 是否需要拆子项目
- 若拆分，子项目名称、目标、依赖、顺序
- 当前优先进入主链的子项目
- 如缺少基础总览，补齐轻量项目简介字段
- 推荐返回 `pm-go` 继续推进的下一动作

若本轮仍处于澄清中，必须明确：

- 当前尚未确认的关键问题
- 为什么这些问题会影响范围判断
- 下一轮只准备确认哪一个问题

若本轮已形成范围结论，必须明确：

- 本次推荐结论
- 推荐理由
- 是否已获得用户确认

## 下一步回流

`pm-scope` 不是独立终点。执行完成后必须把控制权回流到 `pm-go`，由 `pm-go` 继续：

- 记录当前范围判定结果
- 识别当前阶段
- 决定进入 `pm-mm` / `pm-fl` / `pm-ps` / `pm-prd`
- 或在阻塞时停下等待确认

只有在用户确认当前范围结论或拆分方案后，才能回流 `pm-go` 继续主链。

## 状态更新

至少更新：

- `current_stage`
- `scope_decomposition`
- `latest_artifacts.briefs`（如本轮补齐了轻量项目简介）
- `blockers`
- `pending_confirmations`

## 典型触发信号

以下情况应优先触发 `pm-scope`，而不是直接进入后续技能：

- 用户目标仍然模糊，只描述了大方向
- 请求明显覆盖多个模块或多个子系统
- 用户同时要求“先做完整平台”或“把多个大模块一起规划”
- 当前版本边界不清，无法判断什么属于本期
- 怀疑单一 Brief / Feature List / PRD 无法承载清晰边界

## 不负责

- 不写 spec
- 不进入 `writing-plans`
- 不替代 `pm-mm` / `pm-fl` / `pm-prd`
- 不在未确认范围前提前输出下游产物结论
