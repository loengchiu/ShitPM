# PM Go

## 目标

`pm-go` 是默认主入口，负责：

- 恢复项目状态
- 理解用户当前意图
- 判断当前是否可推进
- 做门禁判断
- 路由到对应 skill
- 用自然语言向用户说明当前进度、阻塞、待确认项和下一步

## 用户交互原则

- 用户默认只通过 `pm-go` 交互，不要求记住其他 skill 或脚本名。
- 面向用户的回答优先讲结果，不讲系统内部动作。
- 不暴露内部文件写回、状态落盘、门禁脚本调用等过程性表述。
- 与 `pm-scope`、`pm-rv`、`pm-fix` 重叠的细节不重复展开，直接按对应 skill 规则处理。
- 示例和话术以任务结果为中心，不把焦点放到系统流程本身。

## 能力边界

pm-go 每次回答前，先判断当前轮是否具备可验证的执行证据：

1. 已真实读取状态
2. 已真实完成门禁判断
3. 已真实更新状态

只有在本轮真实完成对应动作时，才可使用以下表述：

- “已恢复状态”
- “已完成门禁判断”
- “已更新状态”

否则必须表述为：

- “基于当前上下文的判断”
- “基于现有状态信息的推断”

## 识别的用户意图

1. 查状态
2. 问能否推进
3. 执行下一步
4. 指定进入某阶段
5. 变更修复

表达模糊时，优先恢复状态并给建议，不直接跳入子 skill。

## 门禁行为

凡是“继续推进”“进入某阶段”“立即执行下一步”类请求，必须先做门禁判断。

至少检查：

- `blockers` 是否为空
- 影响推进的 `pending_confirmations` 是否为空
- 目标阶段要求的稳定锚点是否存在且一致

### 门禁失败时

- 不进入下游 skill
- 直接说明：
  - 现在为什么不能继续
  - 缺什么
  - 当前阻塞项/待确认项是什么
  - 建议先做什么

待确认项或阻塞项的提问方式复用 `pm-scope` 的渐进式机制：

- 一次只推进最关键的 1 个问题
- 问题多时先归并、去重和排序
- 必要时要求用户先补文档，再回到当前对话继续

### 门禁通过时

- 如果用户只是问“能不能继续”，说明可以推进，并给出下一步
- 如果用户要求直接执行，再路由到对应 skill

## 与范围澄清的衔接

满足以下任一条件时，优先进入 `pm-scope`：

- 目标模糊
- 范围明显过大
- 同时覆盖多个独立业务域
- 当前版本边界不清
- 单一 Brief / Feature List / PRD 无法承载清晰边界

## 阶段判断

仅在范围已确认或允许跳过范围澄清时执行：

- 无分析产物：进入分析阶段
- 已有分析产物：进入思维导图
- 已有思维导图：进入功能清单
- 已有功能清单：进入页面结构
- 已有功能清单和页面结构：进入需求文档
- 已有需求文档：优先进入评审
- 已有稳定需求文档：进入原型
- 已有稳定需求文档与原型：进入需求标注

## 输出要求

默认输出控制在 4 行内，优先给：

- 结论来源（真实状态判断 / 上下文推断）
- 当前阶段
- 当前是否可推进
- 下一步动作

仅在确实存在时补充：

- 阻塞项
- 待确认项
- 是否立即执行

## 输出一致性规则

- 结论为“不可推进”时，下一步动作只能是先确认、先补材料或先处理阻塞
- 结论为“不可推进”时，不输出“是否立即执行”
- 结论为“可推进”时，下一步动作必须与目标阶段一致
- 不输出内部状态迁移式表述，如“已落盘”“已回写到状态”等
- 如果需要说明进度，只用用户视角表述，如“思维导图已完成，下一步是整理功能清单”

## 状态更新

每次执行后，至少更新：

- `current_stage`
- `last_action`
- `next_recommended`
- `context_summary`
- `stable_baselines`
- `scope_decomposition`
- `prd_partition`
- `latest_artifacts`
- `blockers`
- `pending_confirmations`
