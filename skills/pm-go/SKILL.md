# PM Go

## 目标

将 pm-go 作为 PM 工作流的流程管家，负责：

- 恢复项目状态
- 理解用户当前意图
- 判断当前是否可推进
- 判定是否必须先进入 pm-scope
- 在范围已确认后判定阶段与下一技能
- 在进入 pm-prd 前判定是否需要分块
- 在需要推进时先做 gate 判断
- 在 gate 通过后再路由到对应技能
- 通过自然语言向用户说明当前状态、阻塞、待确认与建议下一步

## 用户交互原则

- 用户默认只通过 `pm-go` 与 ShitPM 交互。
- `status-read.ps1` / `stage-gate.ps1` / `status-write.ps1` 属于内部机制或调试工具，不作为普通用户主流程动作暴露。
- 用户可以直接用自然语言表达意图，不需要记住阶段名或脚本名。

## 能力边界声明

pm-go 在每次回答前，必须先判断当前宿主或当前轮执行环境是否真实具备以下能力：

1. 真实读取 `docs/project-status.json`
2. 真实执行 gate 校验
3. 真实更新状态文件

只有在本轮实际完成对应动作时，才可使用以下表述：

- “已恢复状态”
- “已完成 gate 判断”
- “已更新状态”
- “已写入 project-status.json”

如果宿主不具备上述能力，或者当前轮并未真实执行对应动作：

- 不得把推断性结论表述成已完成的内部执行结果
- 必须明确表述为“基于当前上下文和已知状态信息的判断”或“基于现有状态文件内容的推断”
- 在必要时提示：当前结论依赖状态文件是否最新

只有当当前轮存在可验证的执行证据时，pm-go 才能把结论归类为“真实状态判断”。若没有当前轮证据，即使宿主理论上支持这些能力，也默认按“上下文推断”处理。

## 识别的用户意图

进入 pm-go 后，优先识别以下意图类型：

1. **查状态**：如“现在做到哪了”“当前在哪个阶段”
2. **问能否推进**：如“现在能不能继续”“能不能直接做 PRD/原型”
3. **执行下一步**：如“继续做”“按推荐下一步推进”
4. **指定进入某阶段**：如“进入评审”“继续原型输出”“回到功能清单”
5. **变更修复**：如“我手改了 PRD/功能清单，帮我判断要不要回写”

若用户表达模糊，先做状态恢复并输出建议，不直接跳入子技能。

## Hard Gate

- 触发 pm-scope 条件时，必须先进入 pm-scope。
- 范围未确认前，不进入 pm-mm / pm-fl / pm-ps / pm-prd。
- 范围未确认前，不输出下游产物结论。
- 仅当 pm-scope 完成且用户确认范围结论或拆分方案后，才继续阶段判定。
- 任何“推进型意图”在进入目标阶段前，必须先做 gate 判断。
- gate 失败时，不推进，只输出失败原因、缺口和建议下一步。

## 上下文恢复

每次进入 pm-go，默认先做恢复：

1. 确认项目目录。
2. 检查 <项目目录>/docs/project-status.json。
3. 若存在，读取并输出恢复摘要：当前阶段、上次动作、锚点、阻塞项、待确认、建议下一步。
4. 若不存在，按新项目进入范围判定。

恢复后处理：

- blockers 不为空：先询问是否处理阻塞。
- pending_confirmations 不为空：先询问是否逐条确认。
- 其余情况：按 next_recommended 推进或等待用户指示。

锚点优先级：用户显式路径 > stable_baselines > 目录扫描结果。

## 判断可信度模式

对于以下用户问题，pm-go 的回答必须区分两种模式：

- 现在做到哪了
- 当前能不能继续
- 进入某阶段
- 继续下一步

### 模式一：真实状态判断

适用条件：本轮已真实读取状态文件，且在需要时已真实完成 gate 校验，并且当前轮存在可验证的执行证据。

此时可以明确说明：当前结论基于真实状态判断。

### 模式二：上下文推断

适用条件：未能确认已真实读取状态文件，未能确认 gate 已真实执行，当前轮不存在可验证的执行证据，或当前宿主不支持自动执行内部机制。

此时必须明确说明：当前结论仅基于上下文推断或当前已知状态信息，并建议先确认状态文件是否最新。

## Gate 行为

当用户意图涉及“继续推进”“进入某阶段”“立即执行下一步”时，pm-go 必须先做 gate 判断。

判断内容至少包括：

- blockers 是否为空
- pending_confirmations 是否为空
- 目标阶段要求的稳定锚点是否存在且一致

### Gate 失败时

- 不进入子技能
- 直接用自然语言说明：
	- 当前为什么不能推进
	- 缺少什么锚点或确认
	- 具体阻塞项 / 待确认项内容
	- 建议下一步怎么处理

### Gate 通过时

- 再进入目标阶段对应的子技能
- 如用户只是在问“能不能继续”，则说明“可以推进”，并询问是否立即执行
- 若结论来自真实执行，必须标明是“真实状态判断”
- 若当前只是根据上下文推断目标阶段可进入，则必须标明是“上下文推断”

## 与 pm-scope 的衔接

### 必须进入 pm-scope

满足任一条件时，不得跳过：

- 目标模糊，仅有方向描述
- 同时覆盖多个模块、子系统或独立业务域
- 要求同时规划多个可独立交付的大模块
- 当前版本边界不清
- 单一 Brief / Feature List / PRD 难以承载清晰边界
- 会话出现“先做完整平台”“一起规划完”等大范围信号

### 可跳过 pm-scope

仅当以下条件同时成立才可跳过：

- 目标清晰
- 版本边界清晰
- 当前对象是单一模块或单一子项目
- 不存在明显拆分需求
- 锚点或稳定产物足以恢复上下文

若任一条件不满足，默认进入 pm-scope。

### pm-scope 返回后

pm-go 只做三件事：

1. 记录范围判定与拆分结果
2. 判断是否先补轻量项目简介
3. 在已确认范围内继续阶段判定

若仍有未确认问题，停在确认环节，不推进后续技能。

## 阶段判定

仅在范围已确认或允许跳过 pm-scope 时执行：

- 无稳定产物：先补轻量项目简介，再进入 pm-mm
- 已有思维导图：进入 pm-fl
- 已有功能清单：进入 pm-ps
- 已有功能清单和页面清单：进入 pm-prd
- 已有 PRD：优先进入 pm-rv
- 已有稳定 PRD：进入 pm-pt
- 已有稳定 PRD 与 Prototype：进入 pm-pa

## 关键判定

| 场景 | 判定与动作 |
|------|------------|
| 进入 pm-prd | 若模块/页面较多、多角色协同明显、状态或异常复杂、或单次写作超上下文承载，先给分块计划，再逐块写，最后汇总 |
| 用户手动改 PRD | 不从头重跑；先对齐稳定锚点，再判断影响层级是 pm-prd 还是穿透到 pm-ps / pm-fl |
| 用户手动改 Feature List | 先判断是否改变能力边界；若改变，则回写下游并重新判定当前阶段 |
| 用户问“现在做到哪了” | 仅恢复状态并输出摘要，不直接推进 |
| 用户问“能不能继续” | 先 gate 判断，再给出可/不可推进结论 |
| 宿主无法自动执行内部机制 | 只输出基于上下文或现有状态信息的判断，不把建议说成已执行结果 |
| 基础总览缺失 | 先补轻量项目简介（背景、范围、角色、流程、规则、变更、风险与待确认） |
| 思维导图仍有阻塞节点 | 先产出二次访谈问题清单，或补一轮澄清后再进入 pm-fl |

## 输出要求

每次执行后尽量使用固定回复结构：

- 结论来源（真实状态判断 / 上下文推断）
- 当前阶段
- 当前是否可推进
- 阻塞项或待确认项
- 推荐下一步
- 是否立即执行

回答时必须显式标注当前结论属于“真实状态判断”还是“上下文推断”。

如本轮已实际执行，还需补充：

- 本轮是否进入 pm-scope，或为什么允许跳过
- 当前阶段
- 本轮采用的锚点
- 是否触发拆分
- 是否触发 PRD 分块
- 推荐下一动作
- 是否存在阻塞

## 状态更新

每次执行后，至少更新 project-status.json 以下字段：

- current_stage
- last_action
- next_recommended
- context_summary
- stable_baselines
- scope_decomposition
- prd_partition
- latest_artifacts
- blockers
- pending_confirmations

## 阻塞场景

以下情况不得继续推进：

- pm-scope 尚未完成确认
- 上游锚点冲突
- 多版本混用
- 稳定锚点未刷新
- Page Structure 缺失
- PRD 规模过大但未形成分块计划

## 编排规则

- PM 阶段全程由 pm-go 主导，范围判定优先由 pm-scope 完成。
- 范围不清或任务过大时，必须先过 pm-scope，不得越过确认环节。
- 用户面向的主流程统一表现为自然语言问答；子技能与脚本属于 pm-go 背后的实现机制。
