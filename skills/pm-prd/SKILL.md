# PM PRD

## 作用

使用这个技能生成和维护 PRD。PRD 是这套 PM 工作流里的主规格锚点，后续 Prototype、评审与相关变更都应围绕 PRD 收敛。

使用 `../../templates/prd.md` 作为强约束交付模板。

## 输入

按以下优先级读取：

1. 稳定 Feature List
2. 稳定 Page Structure
3. 会影响范围、优先级、规则表述或验收口径的补充澄清信息
4. 历史 cards / interviews / 旧稿（仅作补充参考，不得覆盖当前稳定锚点）

如存在 `docs/project-status.json`，优先继承：

- `stable_baselines.feature_list`
- `stable_baselines.page_structure`

## 输出

将 PRD 写入工作项目的 `docs/prd/` 目录。

## 规则

- PRD 视为后续 Prototype 和评审的主规格锚点。
- Feature List 负责组织范围与能力边界，PRD 负责最终需求表达和结构定义。
- Page Structure 负责为 PRD 提供稳定的页面与区域事实源。
- 内容来源优先级固定为：`Feature List + Page Structure > 补充澄清信息 > 历史 cards / interviews / 旧稿`
- 历史材料只能补充，不能覆盖当前稳定锚点。
- `上游锚点` 只允许填写本轮直接支撑 PRD 的当前稳定事实源，默认应为当前 `Feature List` 与当前 `Page Structure`。
- `来源材料` 只允许填写本轮真实直接使用的补充材料；若列入，必须说明用途，如术语口径补充、历史背景核对、既有流程校验。
- 如工作项目中同时存在新旧多个版本的 Feature List / Page Structure，只能继承 `stable_baselines` 指向的当前稳定版，不得自行回退引用旧版。
- 必须严格按模板的一、二、三级章节顺序输出，不得自由重组主章节。
- 信息不足时可在对应章节显式标记“待确认”，不得删除、合并或改写模板主骨架。
- 必须使用纯中文表达；除 `ID`、`CSV`、`PDF`、`API`、`Excel`、`TTS`、`ESC` 等行业通用专有名词外，不要中英混杂。
- 必须使用严格层级序号：一级标题用“一、”，二级标题用“（一）”，三级标题用“1.”，四级标题用“（1）”，五级标题用“①”。
- 只描述业务规则、状态机、数据流向和前端表现，不干涉底层技术实现。

## 业务属性继承

Feature List 中的关键业务属性不能在进入 PRD 后丢失，至少按以下规则继承：

- `涉及角色` 必须在 PRD 的“用户与角色”或对应模块说明中体现
- `版本边界` 必须在 PRD 的“范围边界”中体现本期范围、暂不包含和后续规划
- `优先级` 不要求逐项原样抄写进正文，但当它直接影响本期范围、实现顺序或交付边界时，必须在范围或说明中体现
- 页面承载关系必须保留 Page Structure 中已经稳定的页面归属与区域归属

禁止出现以下情况：

- Feature List 已定义模块角色，PRD 中完全消失
- Feature List 已定义版本边界，PRD 中无法判断本期范围
- Feature List 已标出高优先级关键能力，PRD 中被弱化为可有可无的补充项
- Page Structure 已明确的页面承载关系，在 PRD 中完全消失

## 详细需求说明规则

每个模块下的子功能必须包含：

- 位置
- 目标
- 核心功能点
- 功能描述
- 界面承载说明

### 功能描述

按复杂度灵活组织，不必机械套固定小标题，但应覆盖真实存在的场景。必要时包括：

- 文本与数据展示
- 按钮与操作
- 表单与输入
- 列表与加载
- 弹窗
- 异常与降级
- 数据规范
- 状态流转与边界场景

判断标准：

- 如某类场景客观存在，必须落到具体触发条件、界面反馈和结果变化
- 不允许只写“需支持”“需考虑”“按规范处理”

### 数据规范

如当前功能涉及关键数据规则，应在功能描述中明确：

- 数据来源
- 字段口径
- 展示格式
- 枚举或状态映射
- 脱敏 / 隐藏规则
- 同步、导入导出或一致性规则

### 界面承载说明

必须说明：

- 主承载界面
- 关联界面
- 关键区域
- 承载要点

并且所有界面与区域都必须能回溯到当前 `Page Structure`。

若 `Page Structure` 中没有，不能脑补，只能：

- 标记“待确认”
- 或回退修正 `pm-ps`

### 角色与范围落位

为避免上游业务属性丢失，PRD 在生成时还应满足：

- 模块级或场景级内容应能回溯到 `Feature List` 的模块和涉及角色
- 本期能力范围应能回溯到 `Feature List` 的版本边界
- 页面承载关系应能回溯到 `Page Structure`，并与思维导图中的业务闭环保持一致
- 若当前正文只覆盖 PRD 的一个分块，应明确该分块覆盖的模块 / 场景范围，而不是假装覆盖整篇

## 图表规则

- 图表统一使用 Mermaid
- 顺序图：`sequenceDiagram`
- 状态图：`stateDiagram-v2`
- 业务流程与判断：`flowchart`
- 需要补图时，直接将 Mermaid 代码块放进正文

当满足以下任一条件时，必须补图：

- 状态节点不少于 3 个
- 存在多角色协同、系统间顺序交互或异步链路
- 存在两个及以上条件分支

## 分块生成

如 `pm-go` 或当前上下文判断需要分块：

- 先输出 PRD 分块计划
- 再逐块生成
- 再汇总

不得在规模过大时直接整篇硬写。

## 状态更新

- 更新 `latest_artifacts.prd`
- 如本轮 PRD 已成为稳定基线，刷新 `stable_baselines.prd`
- 如发现 `feature_list` 或 `page_structure` 稳定锚点滞后，先修正状态文件，再继续

## 完成定义

- 已严格按模板主骨架输出
- 详细需求说明已覆盖真实存在的关键展示、操作、输入、加载、弹窗、异常和数据规则
- 界面承载说明中的页面与区域全部可回溯到 Page Structure
- 已继承 Feature List 的角色、范围与关键业务属性
- 复杂流程或状态已补 Mermaid 图
- 如规模过大，已先给出分块计划而不是直接整篇硬写

## 错误处理与回退

- 若稳定 Feature List 或 Page Structure 不足以支撑 PRD，回退到 `pm-fl` 或 `pm-ps`
- 若工作项目存在多版本上游且稳定基线未更新到最新确认版，停止生成并先修复状态锚点
- 若用户要求跳过关键需求说明、异常处理或验收标准，说明会直接削弱 Prototype 和评审质量
- 若界面承载说明中出现 Page Structure 不存在的区域、入口或模块，先删除猜测内容并回退修正事实源
- 若 PRD 与已确认范围发生重大冲突，停止交接并先修复事实上游

## 交接

- 将 PRD 交给 `pm-rv` 或 `pm-pt`
- 保持 PRD 的细节足以约束原型范围、关键流程和后续需求变更
