# PM PRD

## 目标

生成和修订可被研发、测试和设计直接使用的 PRD。

## 上游锚点

- `pm-fl` 产物：能力锚点
- `pm-ps` 产物：页面锚点

如存在 `docs/project-status.json`，优先继承：

- `stable_baselines.feature_list`
- `stable_baselines.page_structure`

## 内容来源优先级

固定为：

`Feature List + Page Structure > 补充澄清信息 > 历史 cards / interviews / 旧稿`

规则：

- 历史材料只能补充，不能覆盖当前稳定锚点
- `上游锚点` 只允许写本轮直接支撑 PRD 的当前稳定事实源
- `来源材料` 只允许写本轮真的直接使用的补充材料

## 强约束

- 必须严格按模板章节顺序输出
- 不得自由重组主章节
- 若信息不足，可标记“待确认”，不得改写骨架
- 必须使用纯中文表达
- 简单功能可以简洁，但必须说透
- 复杂功能必须把状态、分支、失败路径和边界场景写清

## 业务属性继承

`Feature List` 中的关键业务属性不能在进入 PRD 后丢失，至少应按以下规则继承：

- `涉及角色`：必须在 PRD 的“用户与角色”或对应模块说明中体现
- `版本边界`：必须在 PRD 的“范围边界”中体现本期范围、暂不包含和后续规划
- `优先级`：不要求逐项原样抄写进正文，但当优先级直接影响本期范围、实现顺序或交付边界时，必须在范围或说明中体现

禁止出现以下情况：

- `Feature List` 已定义模块角色，但 PRD 中完全消失
- `Feature List` 已定义版本边界，但 PRD 中无法判断本期范围
- `Feature List` 已标出高优先级关键能力，但 PRD 中被弱化为可有可无的补充项

## 详细需求说明规则

每个功能项必须包含：

- 位置
- 目标
- 核心功能点
- 功能描述
- 界面承载说明

### 功能描述

按复杂度灵活组织，不必机械套固定小标题，但应覆盖真实存在的场景，必要时包括：

- 文本与数据展示
- 按钮与操作
- 表单与输入
- 列表与加载
- 弹窗
- 异常与降级
- 数据规范

判断标准：

- 如果某类场景客观存在，必须落到具体触发条件、界面反馈和结果变化
- 不允许只写“需支持”“需考虑”“按规范处理”

### 数据规范

如当前功能涉及关键数据规则，应在功能描述中明确：

- 数据来源
- 字段口径
- 展示格式
- 枚举或状态映射
- 脱敏 / 隐藏规则
- 同步、导入导出或一致性规则

### 界面承载说明

必须说明：

- 主承载界面
- 关联界面
- 关键区域
- 承载要点

并且所有界面与区域都必须能回溯到当前 `Page Structure`。

若 `Page Structure` 中没有，不能脑补，只能：

- 标记“待确认”
- 或回退修正 `pm-ps`

### 角色与范围落位

为避免上游业务属性丢失，PRD 在生成时还应满足：

- 模块级或场景级内容应能回溯到 `Feature List` 的模块和涉及角色
- 本期能力范围应能回溯到 `Feature List` 的版本边界
- 当前正文若只覆盖 PRD 的一个分块，应明确该分块覆盖的模块 / 场景范围，而不是假装覆盖整篇

## 图表规则

- 图表统一使用 Mermaid
- 顺序图：`sequenceDiagram`
- 状态图：`stateDiagram-v2`
- 业务流程与判断：`flowchart`
- 需要补图时，直接将 Mermaid 代码块放进正文

## 分块生成

如 `pm-go` 或当前上下文判断需要分块：

- 先输出 PRD 分块计划
- 再逐块生成
- 再汇总

不得在规模过大时直接整篇硬写。

## 状态更新

- 更新 `latest_artifacts.prd`
- 如本轮 PRD 已成为稳定基线，刷新 `stable_baselines.prd`
- 如发现 `feature_list` 或 `page_structure` 稳定锚点滞后，先修正状态文件，再继续
