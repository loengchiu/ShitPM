# PM RV

## 目标

检查 `Mindmap`、`Feature List`、`Page Structure` 和 `PRD` 的单层质量与跨产物一致性，判断是否需要：

- 回写上游
- 回退阶段
- 修正稳定锚点

支持两种评审模式，根据用户意图自动判定。

## 模式判定

- **自检模式**：用户未提及具体角色，AI 独立完成一致性检查
- **多角色评审模式**：用户提及角色、评审会或开发对齐，切换到多角色视角

## 自检模式

### 输入优先级

1. 用户显式给出的当前文件路径
2. `stable_baselines`
3. 目录扫描结果

### 评审对象判定

- 用户明确指定对象时，按指定对象评审
- 用户只说“开评审”时，按当前阶段或最近产物判定
- 无法判清时，先要求确认对象，不默认按 PRD 处理

默认前置关系：

- `Mindmap`：至少存在 `latest_artifacts.mindmap` 或显式路径
- `Feature List`：至少存在 `latest_artifacts.feature_list`、`stable_baselines.feature_list` 或显式路径
- `Page Structure`：至少存在 `latest_artifacts.page_structure`、`stable_baselines.page_structure` 或显式路径
- `PRD`：至少存在 `latest_artifacts.prd`、`stable_baselines.prd` 或显式路径

### 评审范围

- `mind`
- `feat`
- `page`
- `prd`

### 必做检查

#### 1. 版本关系

- 显式路径与稳定锚点是否一致
- 是否存在新版本已生成但稳定锚点仍指向旧版

#### 2. 产物一致性

- `Mindmap` 是否仍代表当前业务范围、主流程、关键判断点和系统结构
- `Feature List` 是否仍代表当前能力边界
- `Page Structure` 是否仍代表当前页面承载边界
- `PRD` 是否越权重新定义能力或页面结构

#### 3. 分层评审结论（质量层推进建议）

这部分不是 runtime gate。  
`stage-gate` 负责“能不能进”，这里负责“值不值得进”。

- `mind`：范围、主流程、模块、系统结构、关键判断点、状态分支、待澄清问题是否足以支撑 `feat`
- `feat`：模块、功能点、角色、版本边界是否足以支撑 `page`
- `page`：页面目标、区域承载、关键操作与关键状态是否足以支撑 `prd`
- `prd`：规格表达、异常处理、数据规则和交互逻辑是否足以支撑后续原型与评审

评审 `prd` 时，对交互逻辑采用按复杂度判断的口径：

- 简单交互：只要触发、反馈和结果清楚即可，不要求展开成长段说明
- 中等复杂交互：应能看出交互顺序、关键状态变化和必要分支
- 高复杂交互：才要求把触发条件、交互顺序、反馈方式、状态变化和关键分支写透

若模块内存在复杂表单，还应检查表单相关的数据规范是否直接写在该模块的详细需求说明里，而不是遗漏或散落到模块外其他位置。

#### 4. 回写判定

- 只影响规格表达：留在当前稿件
- 影响业务流程、判断节点、状态分支或系统结构：回写 `mind`
- 影响页面承载：回写 `page`
- 影响能力边界：回写 `feat`

#### PRD 内部一致性检查

当评审对象包含 PRD 时，除现有检查外，增加以下检查项：

1. 功能覆盖检查
   - Feature List 中每条"本期"功能点是否在 PRD 中有对应描述
   - PRD 中是否有 Feature List 中不存在的功能点（越权新增）

2. 页面覆盖检查
   - Page Structure 中每个页面是否在 PRD 第 5 章有对应章节
   - PRD 中是否有 Page Structure 中不存在的页面

3. 字段一致性检查
   - 同一字段在不同页面的展示格式是否一致
   - 同一字段在不同页面的校验规则是否一致
   - 数据字典中的枚举值是否和页面说明中的选项一致

4. 权限一致性检查
   - 权限矩阵中的规则是否和各页面说明中的按钮可见性一致
   - 不同角色看到的内容是否和权限矩阵匹配

5. 状态一致性检查
   - 状态流转图中的状态是否和各页面中的状态筛选/展示一致
   - 状态变更触发条件是否和横切规则中的通知规则一致

6. 交叉引用完整性检查
   - 所有 FL-__ 引用是否指向 Feature List 中的真实条目
   - 所有 PS-__ 引用是否指向 Page Structure 中的真实条目
   - 所有 DD-__ 引用是否指向数据字典中的真实条目

7. 场景覆盖检查
   - 每个区域的 9 类关键场景（展示、操作、输入、加载、弹窗、异常、数据规范、交互逻辑、状态流转与边界场景）是否覆盖了真实存在的场景
   - 不存在的场景类别允许省略，但存在的必须写清楚

8. 模糊表述检查
   - 全文搜索以下禁用词，确保零命中："需支持"、"需考虑"、"按规范处理"、"详见原型"、"同常规"、"待定"
   - 发现命中时标注位置，要求修改为具体描述

9. 边界场景检查
   - 涉及数值输入的区域是否标注了边界值
   - 涉及列表的区域是否标注了空状态和数据量上限
   - 涉及状态流转的功能是否标注了临界状态
   - 涉及并发操作的功能是否标注了冲突处理方式

### 冲突优先级

- 如果评审团意见与稳定锚点、正式 skill 规则或对应合同冲突，以稳定锚点和正式规则为准。
- 评审团可以指出风险、要求解释、建议回写，但不能直接覆盖事实源。
- 只有当评审结论被确认并进入 `fix` 或对应上游修正后，事实源才算变化。

### 输出要求

必须输出：

- 实际采用的当前路径
- 与 `stable_baselines` 的关系
- 当前问题属于哪一层
- 是否需要回写
- 是否需要回退
- 推荐下一步动作

写入 `docs/review-meetings/YYYY-MM-DD_review-<topic>.md`，并更新 `latest_artifacts.reviews`。

## 多角色评审模式

多角色评审不是表演式角色扮演，而是按不同评审视角和评审团组合检查同一产物。

### 角色与默认关注域

| 角色 | 关注域 |
|------|--------|
| 业务经理 | 业务目标、版本价值、范围取舍 |
| 项目经理 | 排期依赖、风险、跨团队协作 |
| 产品经理 | 闭环、边界、异常与规则 |
| 前端 | 交互复杂度、状态、组件边界 |
| 后端 | 实体、接口、权限、状态机 |
| UI | 信息层级、视觉承载、实现一致性 |
| 测试 | 可测试性、边界、异常、回归风险 |

### 评审团组合模板

- `需求边界评审团`：产品经理、业务经理、项目经理
- `页面承载评审团`：产品经理、前端、UI
- `可开发性评审团`：产品经理、前端、后端、测试
- `上线风险评审团`：项目经理、产品经理、后端、测试

### 结论判定

- `通过`：与上游锚点一致，无关键风险
- `带条件通过`：主体成立，但仍有少量待处理项
- `阻塞`：范围、页面承载、事实源或实现前提存在根本冲突

## 评审后问答模式

若评审结束后存在需要用户回答的待处理问题，自动进入问答模式：

- 不一次性把所有问题丢给用户
- 逐条收敛，默认每轮只推进最关键的 1 个问题；强相关时最多同轮推进 3 个
- 问题较多时先归并、去重并按阻塞度排序
- 当前问题回答后，重新判断哪些问题已自然消失、哪些仍需继续问

建议阈值：

- `1-3` 个问题：可逐条问答
- `4-7` 个问题：先归并后逐条问答
- `8+` 个问题：先输出问题地图与首要问题，不允许一次性全量追问

### 问题定性

每条待处理问题先定性：

| 定性 | 含义 | 处理方式 |
|------|------|---------|
| **真实风险** | 必须解决，否则阻塞推进 | 写入 `blockers`，路由处理 |
| **过度担忧** | 影响可接受或已有兜底 | 记录说明，不阻塞 |
| **待观察项** | 现在无法定论，需后续跟进 | 写入 `pending_confirmations` |

无法判定时，默认先按 **真实风险** 处理。

### 与 fix 的联动

如果评审问答后的答案已经改变事实源，不能只停留在评审记录里，必须继续路由到 `fix`。

典型触发场景：

- 用户回答导致当前稿件不再只是文案修订，而是改变了稳定锚点关系
- 用户回答确认某问题确实影响 `Mindmap`、`Feature List`、`Page Structure` 或 `PRD`
- 当前结论已足以确定需要正式回写上游

`rev` 负责发现并定性问题；`fix` 负责正式处理变更。

## 前置检查

执行前先读取 `docs/project-status.json`。
1. 如果文件不存在，停止执行并告知用户："当前项目尚未初始化 ShitPM，请先输入'初始化 ShitPM'。"
2. 检查 `blockers` 是否为空，不为空则停止并告知用户。
3. 检查 `pending_confirmations` 是否为空，不为空则停止并告知用户。
4. 检查 `stable_baselines` 中对应的上游锚点文件是否存在，不存在则停止并告知用户。
5. 若未满足，不得继续推进。
