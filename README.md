# ShitPM

ShitPM 是一个面向 AI IDE（如 GitHub Copilot、Codex、Cursor 等）的 PM 工作流增强工具。它不是独立应用，而是一套运行在宿主里的阶段化方法、状态约束、模板和技能系统，用来把需求从原始资料逐步收敛到分析简报、思维导图、功能清单、页面结构、PRD、原型和评审结果，尽量避免 AI 在长链路生成中常见的发散、越权和锚点漂移。

## ✨ 核心特性

- **纯自然语言优先**：对普通用户来说，默认主入口只有 `pm-go`。你只需要描述当前意图，系统会优先结合当前状态和上下文判断阶段、解释卡点并推荐下一步。
- **阶段化产物链**：主链明确分为分析简报、思维导图、功能清单、页面结构、PRD、原型、原型注释和评审，不再把所有内容混成一份大文档。
- **状态与稳定锚点**：底层通过 `docs/project-status.json` 维护 `latest_artifacts`、`stable_baselines`、`blockers`、`pending_confirmations` 等状态，尽量让下游只消费已经稳定的事实源。
- **运行时门禁**：从 `pm-ps` 开始，进入关键阶段前可通过 `stage-gate.ps1` 校验上游 stable baseline 是否真实存在、是否仍有阻塞项或待确认项。
- **变更回写与留痕**：`pm-fix` 不只判断是否要回写 `Feature List` 或 `Page Structure`，若已自动改动其他生成物，还要求同步留下修正记录。

## 🧭 当前主链

当前 ShitPM 的主链是：

`pm-go -> pm-scope -> pm-analysis -> pm-mm -> pm-fl -> pm-ps -> pm-prd -> pm-rv -> pm-pt -> pm-pa`

这条主链的设计重点不是“尽快一口气跑完”，而是逐层收敛：

1. 先把原始资料整理成正式分析产物
2. 再把分析结果结构化成思维导图
3. 再把导图压成稳定能力边界
4. 再把能力落到页面承载
5. 最后才展开成 PRD、原型和评审结果

## 🧱 阶段产物与衔接

| 阶段 | 生成物 | 解决的问题 | 交给下一步什么 |
| --- | --- | --- | --- |
| `pm-analysis` | 项目简报 | 原始资料里哪些是已确认事实、哪些是推断、哪些仍冲突 | 正式分析产物 |
| `pm-mm` | 思维导图 | 业务范围、主流程、模块结构、关键判断点是否清楚 | 已收敛导图 |
| `pm-fl` | Feature List | 这次版本到底做哪些能力、哪些不做 | 稳定能力边界 |
| `pm-ps` | Page Structure | 这些能力分别由哪些页面和区域承载 | 稳定页面事实源 |
| `pm-prd` | PRD | 如何把流程、能力、页面承载展开成可执行规格 | 主规格文档 |
| `pm-rv` | Review 结果 | 当前链路是否一致、是否需要回写或回退 | 结论、阻塞项、待确认项 |
| `pm-pt` | HTML 原型 | 如何把稳定 PRD 可视化表达出来 | 可评审原型 |
| `pm-pa` | 原型注释 | 如何让原型可评审、可开发、可交接 | 带注释的交付层 |

## 🛠️ 各 Skill 的特点

- `pm-go`：流程管家。负责状态恢复、可推进性判断、下一步推荐、指定阶段进入和子技能路由。
- `pm-scope`：范围探索器。用于先问清目标、范围和是否需要拆子项目。
- `pm-analysis`：正式分析产物阶段。把原始资料整理成项目简报，沉淀资料分析结论、关键流程文字版、二次访谈补充与修正、变更留痕。
- `pm-mm`：结构化思维导图阶段。负责把分析结果变成范围、主流程、模块、系统结构和待澄清问题，并支持二次澄清回写。
- `pm-fl`：功能清单阶段。把导图收敛成稳定能力边界，强调模块、功能、版本边界和涉及角色，不混入页面结构。
- `pm-ps`：页面承载阶段。把能力映射到页面与区域，是 PRD 和 Prototype 的共同页面事实源。
- `pm-prd`：规格展开阶段。以 `Feature List + Page Structure` 为双锚点，生成真正可执行、可评审的 PRD。
- `pm-rv`：评审阶段。支持 AI 自检和多角色评审，能判断一致性问题、回写建议、回退层级，以及写入 `blockers / pending_confirmations`。
- `pm-fix`：定向修复阶段。用于用户手工改稿后的影响范围判定、回写建议、稳定锚点修正和修正记录留痕。
- `pm-pt`：高保真原型阶段。以稳定 PRD 为事实源输出 HTML 原型，并继承统一视觉基线。
- `pm-pa`：原型注释阶段。把 PRD 与原型拼成真正可评审、可交付的标注层。

## 🏗️ 目录结构概述

- `skills/`：PM 核心工作流环节的运转逻辑与 Prompt 定义。
- `commands/`：短命令快捷触发入口（如 `pm-go`, `pm-fl` 等）。
- `templates/`：各环节产物生成的专属 Markdown/HTML 标准模板。
- `contracts/`：上下游工作流节点间必须严格遵守的契约卡点。
- `installers/`：针对各不同 IDE 宿主环境的自动化辅助安装脚本。
- `docs/`：包含白皮书、详细使用手册（`MANUAL.md`）及项目状态机制存储。

## 📦 安装与快速开始

在当前工作区目录下，打开支持的终端，执行以下脚本即可完成初始化部署：

```powershell
powershell -File .\installers\install.ps1
```

该脚本将自动探测你当前已安装的协同 AI 工具，一键完成安装和关联。

安装完毕后，想要立刻启动项目，请向你的 AI 助手（IDE 对话框中）输入：
```text
/pm-go 老板让我在首页加个签到日历。
```

从这里开始，你后续也仍然只需要继续对 `pm-go` 说人话，例如：

```text
/pm-go 现在做到哪个阶段了？
/pm-go 当前能不能继续推进？
/pm-go 继续下一步。
/pm-go 我刚手改了 PRD，帮我判断要不要回写。
```

## 🛠️ 使用方式

对普通用户来说，推荐口径只有一条：**始终从 `pm-go`进入。**

`pm-go` 会优先在背后完成这些判断：

1. 恢复当前项目状态
2. 判断当前是否可推进
3. 识别你是在查状态、继续推进、进入某阶段，还是做变更修复
4. 在需要推进时先做门禁判断
5. 条件满足时再路由到对应能力

正常情况下，你可以把它理解成“只管说人话，剩下交给 `pm-go`”。  
但也需要知道一条边界：这些结论是否来自**真实状态判断**，仍然取决于当前宿主是否真的完成了状态恢复、门禁校验和状态维护，并且当前轮有可验证的执行证据；如果没有，系统应明确告诉你当前结论更接近基于上下文或现有状态信息的判断。

### 你真正需要记住的只有这几种问法

- `pm-go 现在做到哪个阶段了？`
- `pm-go 当前能不能继续推进？`
- `pm-go 继续下一步。`
- `pm-go 进入评审阶段。`
- `pm-go 我刚手改了 PRD，帮我判断要不要回写。`

### 背后的能力仍然存在，但不要求你手记

ShitPM 仍然保留完整链路能力：范围澄清、脑图、功能清单、页面结构、PRD、评审、原型、原型批注、变更修复。

只是现在这些能力默认由 `pm-go` 在背后调度，你不需要再把它们当成普通用户的主流程命令来记忆。

## 🚧 当前设计边界

- ShitPM 的目标宿主是 AI IDE，而不是独立运行应用。
- 自然语言是主交互方式，但“当前结论是不是来自真实状态判断”，仍取决于宿主是否真的完成了状态恢复、门禁校验和状态维护，并且当前轮有可验证的执行证据。
- `latest_artifacts` 代表本轮产出，`stable_baselines` 代表已锁定事实源；两者不能混用。
- 从 `pm-ps` 开始，若仍存在 `blockers` 或 `pending_confirmations`，运行时门禁会阻止继续推进。
