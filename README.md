# ShitPM

> 让 AI IDE 输出工程级 PM 文档，而不是聊天级草稿。

直接让 AI 写 PRD，你大概遇到过这些问题：

- 写到后面，AI 忘了前面说过什么，上下文越长越崩
- 改了一个字段，PRD 里三处对不上，自己都没发现
- 功能清单和页面结构各写各的，评审时才发现对不齐
- 让 AI "重新生成"，结果推翻重来，之前的讨论全白费

ShitPM 解决这些问题。它不是另一个"帮我写 PRD 的 prompt"，而是一套**运行在 AI IDE 里的 PM 工作流系统**——把产品从需求到交付拆成 9 个阶段，每一步都有规则约束、质量门禁和一致性校验。

---

## 它怎么做到的

### 不会越写越乱

AI 的上下文窗口有限，信息越多质量越差。ShitPM 把整个 PM 流程拆成 9 个阶段，每个阶段只处理当前需要的信息。思维导图太大？自动拆块生成再合并。原型页面太多？逐页生成，不撑爆上下文。

你不需要管理上下文——ShitPM 替你管。

### 不会在错误基础上继续

每个阶段推进前，ShitPM 会自动检查：前置产物是否就绪、有没有未解决的问题、上一版是否已确认。不满足条件，不允许往下走。

与其在错误的基础上堆砌 10 页文档，不如在第 2 页就停下来。

### 不是让 AI 自由发挥

每个阶段都有明确的规则：输入什么、输出什么、自检什么。生成完 PRD，AI 会自己跑 9 项一致性检查——功能覆盖、字段一致、权限匹配、交叉引用……发现问题直接标出来，不等你发现。

评审时，你可以让 AI 模拟不同角色（前端、后端、测试、业务）分别审视同一份文档，逐条收敛问题，而不是一次性丢给你一堆意见。

### 改一处，处处同步

从需求简报到 PRD，关键业务字段会逐层传递——思维导图里的判断点、功能清单里的关联字段、页面结构里的字段引用、PRD 数据字典里的定义，每一层都会检查上一层的字段是否被完整承接。

改了 PRD 里的一个校验规则？ShitPM 会自动追溯：上游哪些产物提到了这个字段，需不需要同步修改，影响范围是局部还是整体。不会出现"改了这里忘了那里"的情况。

## 主链

```
范围 → 简报 → 思维导图 → 功能清单 → 页面结构 → PRD → 评审 → 原型 → 原型注释
```

| 阶段 | 产物 | 做什么 |
|------|------|--------|
| `scope` | 范围结论 | 判断做什么、不做什么、要不要拆项目 |
| `sum` | 项目简报 | 把散乱的需求碎片整理成结构化输入 |
| `mind` | 思维导图 | 收敛业务流程、模块划分和关键判断点 |
| `feat` | 功能清单 | 固定能力边界和版本范围，防止需求蔓延 |
| `page` | 页面结构 | 固定页面和区域承载，明确"做什么页面" |
| `prd` | PRD | 展开为可执行规格，带数据字典和交互逻辑 |
| `rev` | 评审结果 | 跨产物一致性检查，发现问题或放行 |
| `mock` | 交互原型 | 把 PRD 变成可在浏览器里点的原型 |
| `note` | 原型注释 | 给原型加上开发说明，形成可评审交付层 |

---

## 特色能力

### 🔍 rev — 不只是"检查一遍"

两种模式按需切换：

- **自检模式**：AI 独立完成 10 项一致性检查，从功能覆盖到字段传递，逐项过
- **评审团模式**：指定角色组合（产品+前端+后端+测试），多视角评审同一份文档

评审后发现问题？逐条收敛，每轮只推进最关键的 1 个问题，不会一口气丢给你 20 个待办。

### 🔧 fix — 改了就改对

手动改了已确认的产物？ShitPM 不会直接忽略，也不会盲目重写：

1. 先分析影响范围——改了什么、影响哪些产物、局部还是整体
2. 再判定回写粒度——只改受影响的章节，不动其他内容
3. 最后处理冲突——如果改动和已确认的内容矛盾，标记出来让你确认，不自动覆盖

每一步都留记录，随时可以追溯"为什么改、改了谁"。

### 🖼️ mock — 能点的原型，不是截图

生成的原型是带交互的 HTML 页面，不是静态图片。按钮能点、表单能填、弹窗能弹。评审时直接在浏览器里操作，不用脑补交互效果。

## 使用场景

### 场景 1：从零做一个新功能

老板说要做供应商准入审批流程。你打开 AI IDE：

```
/start
老板让我做一个供应商准入审批流程，需要支持多级审批、资质到期提醒、供应商黑名单
```

然后按节奏推进：

```
/sum        ← 整理你的原始资料和需求碎片
/mind       ← 生成思维导图，收敛流程和模块
/feat       ← 生成功能清单，固定能力边界
/page       ← 生成页面结构
/prd        ← 生成完整 PRD
/rev        ← 自动做一致性评审
/mock       ← 生成交互原型
```

每一步都有门禁检查，上一步没做好，下一步不会开始。

### 场景 2：评审发现问题，回写上游

评审 PRD 时发现功能清单漏了一个模块：

```
/rev
对当前 PRD 做评审
```

AI 输出评审结论：功能清单缺少"供应商资质续期"模块，建议回写。

```
/fix
按评审结论回写功能清单，补充资质续期模块
```

AI 自动分析影响 → 判定回写范围 → 检查冲突 → 执行回写 → 留修正记录。你不需要手动同步其他文档。

### 场景 3：改了一个字段，追溯影响

你手动改了 PRD 里"供应商名称"的校验规则（从无限制改为 50 字符上限）：

```
/fix
我改了 PRD 数据字典里供应商名称的校验规则
```

AI 自动追溯：这个字段在思维导图、功能清单、页面结构里有没有出现？需不需要同步修改？影响范围是局部还是整体？只改该改的地方，不动其他内容。

### 场景 4：拉团队一起评审

需求快定稿了，要拉前端和后端一起过：

```
/rev
用可开发性评审团（产品经理、前端、后端、测试）评审当前 PRD
```

AI 分别从前端视角（交互复杂度、状态管理）和后端视角（实体、接口、权限）给出评审意见。问题逐条收敛，不一次性全丢给你。

## 安装

详见 [INSTALL.md](INSTALL.md)。

最小上手步骤：

1. 安装 ShitPM
2. 在你的业务项目中执行 `/start`
3. 用自然语言描述需求，或用短命令推进阶段

```
/start /sum /mind /feat /page /prd /rev /mock /fix /note
```
