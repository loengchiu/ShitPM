# PRD Generation Rules

本文件只定义 `prd` 阶段的生成规则。进入生成步骤时读取；不要把自审、门禁、完成定义混在这里。

## 文档主骨架

- 标题层级必须遵循：一级标题用“一、”，二级标题用“（一）”，三级标题用“1．”，四级标题用“（1）”，五级标题用“①”
- 文档主骨架固定为：`文档概述 / 范围 / 用户角色与权限 / 数据字典 / 详细需求说明`
- `跨页面规则 / 系统依赖 / 非功能要求 / 验收标准汇总 / 建议实施顺序 / 风险与待确认 / 附录` 为按需章节
- 按需章节仅在形成真实约束时出现，不得为凑完整度硬写
- 信息不足时，对主骨架中的相关位置可显式标记“待确认”；按需章节若无真实约束则整节不出现

## AI锚点区

- 给 AI / 给流程系统看的链路信息，统一收口到 PRD 最后一个章节 `AI锚点区`
- 正文中不分散堆放以下信息：
  - 上游锚点
  - 来源材料
  - 页面结构引用（PS-__）
  - 功能点编号（FL-__）
  - PRD 章节映射
  - 字段链路来源
  - 字段线索去向说明
- 页面正文顶部可保留“关联功能点”，但只写功能点名称，不写功能点编号

## 数据字典收敛规则

- 字段线索公共约束遵循 `contracts/field-clue-rules.md`
- 先汇总 `sum + mind + feat + page` 的字段线索，形成完整字段候选清单，再补充正式规则定义
- 字段候选清单中的线索允许重名、近义名、不完整描述；`prd` 负责统一口径并收敛为正式字段
- 数据字典正文中的“业务来源”只写业务可读口径，如：`用户填写 / 系统生成 / 外部同步 / 关联带出`
- 不得把 `上游线索收敛 / PRD 阶段新增` 这类链路来源写进正文表格
- 每个正式字段都要标明链路来源：`上游线索收敛` 或 `PRD 阶段新增`，但该信息统一写入文末 `AI锚点区`
- 若字段为 `PRD 阶段新增`，必须说明新增理由
- 若上游线索未进入数据字典，也必须说明其去向：`已合并 / 无效线索 / 待确认`

## 页面组织规则

- 第 5 章按页面组织，每个页面对应 Page Structure 中的一个页面
- 页面顺序与 Page Structure 中的定义一致
- 页面内按区域描述，区域定义来自 Page Structure
- 页面正文中的关联功能点优先写功能点名称；功能点编号统一沉淀到 `AI锚点区`
- 不允许出现 Page Structure 中不存在的页面或区域
- 不允许出现功能清单中不存在的功能点

## 表达优化与结构化写法

- 页面章节优先按业务语义组织，如：页面目标、页面构成、查询与列表、核心操作、业务约束、交互约定
- `页面目标` 默认控制在一句话，只说明该页面存在的目的，不拆职责清单
- 复杂页面保留结构化表达，简单页面允许收束，避免所有页面都使用同一信息密度
- 页面说明要合理混用三种表达：
  - 自然段：承接说明
  - `1. 2. 3.`：顺序流程、审核链路、操作步骤
  - `·`：并列构成、展示项、规则项、约束项
- 不得把原本应使用 `1. 2. 3.` 或 `·` 的内容全部压成普通段落
- 通用 B 端规则（如默认分页、通用 loading、常规空状态）除非存在特殊要求，否则不展开成大段正文
- 不得为了显得完整而补写空洞栏目；某类场景不存在时可以不单列
- 不得使用过强的模板腔表达，如反复出现“用于承载”“用于支撑”“需支持”“按规范处理”“同常规”
- 除标题外尽量少用加粗，除非该信息确实是读者必须第一眼锁定的核心约束
- 非页面章节不追求篇幅对称；能用 2-3 行写清的，不要扩成整段“背景说明”或“方案说明”

## 关键场景覆盖要求

按复杂度灵活组织，但必须覆盖真实存在的关键场景。以下 9 类场景只要真实存在就必须写，不存在的不要硬凑：

1. 展示：数据怎么展示、展示格式、排序规则、空状态展示内容
2. 操作：按钮/链接的触发条件和执行结果
3. 输入：输入方式、校验规则、默认值、格式限制、输入长度
4. 加载：首次加载、分页加载、搜索加载、加载中状态、空数据状态、加载失败状态
5. 弹窗：弹窗/抽屉/确认框的触发条件、弹窗内容、操作项、关闭行为
6. 异常：权限不足提示、数据异常、网络异常、超时、并发冲突
7. 数据规范：字段展示格式、单位、脱敏规则、长文本截断
8. 交互逻辑：按复杂度分层写，作为功能描述的一部分，不单独成章
9. 状态流转与边界场景：状态变更触发条件、边界值、临界状态

## 页面结构化写法约束

- 页面构成、展示字段、筛选条件、并列约束、并列规则，优先使用 `·`
- 新增、编辑、删除、提交、审核、确认、回写等存在先后顺序的动作，优先使用 `1. 2. 3.`
- 审核链路、多节点流转、分支处理，不得只写成一句概括，必须拆成编号步骤或“正常路径 / 驳回路径 / 中断条件”
- 只有解释背景、补充上下文或收束结论时，才使用纯自然段

## 交互逻辑分层写法

- 简单交互：1-2 句写清触发、反馈和结果
- 中等复杂交互：补充交互顺序、关键状态变化和必要分支
- 高复杂交互：细写触发条件、交互顺序、反馈方式、状态变化、关键分支和中断条件

判断标准：
- 只有一个触发和一个结果：简单
- 有校验或 2-3 个顺序步骤：中等
- 有状态机、多方协作、多个分支或中断条件：高复杂

## 禁用模糊表述

每条描述必须写清触发条件、系统行为和用户可见结果，禁止：

- `需支持 / 需考虑`
- `按规范处理 / 同常规`
- `详见原型 / 待定`

## 边界场景要求

- 数值输入标注边界值（最大值、最小值、精度）
- 列表标注空状态与数据量上限（如有）
- 状态流转标注临界状态
- 并发操作标注冲突处理方式

## 长文档分块生成协议

当 PRD 预估超过 1000 行时，必须启用分块生成模式。

生成顺序：
1. 数据字典（第 4 章）
2. 横切规则（第 6 章）
3. 权限矩阵（第 3 章）
4. 页面说明（第 5 章），逐页生成
5. 验收标准汇总（第 9 章）
6. 文档概述 + 范围（第 1-2 章）

分块生成循环（每个页面）：
1. 回读上游锚点中与当前页面相关的内容
2. 回读已生成页面的结构化摘要（不读全文）
3. 读取当前页面的 PRD 模板结构
4. 生成当前页面（每个区域按 9 类场景检查覆盖）
5. 保存到文件
6. 生成当前页面的结构化摘要
7. 从上下文中移除当前页面的完整内容，只保留摘要

页面摘要至少保留：页面名称、关联功能点名称、区域列表、关键规则、待确认项。

## 图表规则

- 复杂流程、状态流转、多角色顺序交互、分支审批、系统边界/组件关系，必须补图
- 图表详细风格、命名、生成顺序和回溯门禁，统一遵循 `diagram-style.md`

## 防幻觉规则

- 每个功能点、字段、页面、区域、状态，必须在 Feature List、Page Structure 或数据字典中找到依据
- 找不到依据的内容，必须标记 `[待确认]`，不得自行编造具体值、具体规则、具体交互
- 禁止“合理推断”：上游未声明的能力点、页面、区域、流程分支，不得自行补充
- 允许扩展“怎么做”，不允许扩展“做不做”
