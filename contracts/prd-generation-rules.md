# PRD Generation Rules
本文件只定义 `prd` 阶段的生成规则。进入生成步骤时读取；不要把自审、门禁、完成定义混在这里。

## 文档主骨架

- 标题层级：`##` 章标题、`###` 节标题、`####` 小节标题，最多 3 级
- 主骨架固定：`文档概述 / 范围 / 业务流程 / 用户角色与权限 / 数据字典 / 详细需求说明`
- 按需章节：`公共模块说明 / 系统依赖 / 非功能要求 / 验收标准汇总 / 建议实施顺序 / 风险与待确认 / 附录`，仅在形成真实约束时出现
- 信息不足时主骨架可标"待确认"；按需章节无真实约束则整节不出现
- Markdown 标题层级是正文结构锚点，导出 `docx` 时优先按标题层级映射样式，不靠后处理猜层级
- 标题文本可保留中文编号前缀增强可读性，但不得跳级；推荐顺序：章标题 `一、`，节标题 `（一）`，小节标题 `1．`
- `（1）` 作为正文内第四级顺序表达使用，不新增新的 Markdown 标题层级

## AI锚点区

- 给 AI / 流程系统看的链路信息，统一收口到 PRD 最后一章 `AI锚点区`
- 正文中不出现：上游锚点、来源材料、PS-__/FL-__ 编号、字段链路来源/去向
- 页面正文顶部可保留"关联功能点"名称，不写编号

## 数据字典收敛规则

- 字段线索公共约束遵循 `contracts/field-clue-rules.md`
- 先汇总 `sum + mind + feat + page` 的字段线索形成候选清单，再补充正式定义
- 正文"业务来源"只写：`用户填写 / 系统生成 / 外部同步 / 关联带出`
- 链路来源（`上游线索收敛` / `PRD 阶段新增`）统一写入文末 `AI锚点区`
- PRD 阶段新增字段必须说明理由；上游线索未进入数据字典必须说明去向

## 页面组织规则

- 第 6 章按页面组织，顺序与 Page Structure 一致
- 页面内部按**用户动作**组织，用动词短语作小标题（如"查询入库记录""批量导入"），不按 UI 区域盘点（如"筛选区""列表区"）
- 不允许出现 Page Structure 中不存在的页面、区域或功能点

## 页面内部写法（核心）

### 写作风格

按操作顺序写成**产品规格说明**，语言自然但克制，信息密度高，优先写清页面必须如何展示、如何交互、如何反馈。确保覆盖触发条件、操作过程、操作结果和异常处理。**不要**套用固定格式（如"触发条件：/ 操作过程：/ 操作结果：/ 异常处理："）逐条罗列，也不要写成解释设计思路的聊天口吻。

### 复杂度分级

- 简单（一个触发一个结果）：1-2 句带过
- 中等（有校验或 2-3 步骤）：补操作顺序和关键状态变化
- 高复杂（状态机、多方协作、多分支）：细写触发、顺序、状态、分支和中断

### 正反例

```
❌ 填空味（禁止）：
#### 1．查询入库记录
- 触发条件：进入资产入库页，或修改筛选条件后点击"查询"按钮
- 操作过程：系统按当前筛选条件检索入库记录
- 操作结果：列表刷新展示符合条件的入库记录，默认按申请时间降序排列，每页 20 条
- 异常处理：无匹配结果时展示"暂无入库记录"

✅ 自然叙述（推荐）：
#### 查询入库记录：
- 进入入库页后，系统按当前筛选条件（入库状态、目标服务区、申请时间范围）加载入库记录列表，默认按申请时间降序排列，每页 20 条。
- 无匹配结果时展示"暂无入库记录"。
```

```
❌ 填空味（禁止）：
#### 2．发起入库申请
- 触发条件：服务区管理员点击"新增入库"按钮
- 操作过程：打开入库申请弹窗，选择标准物资，填写数量和税后单价，点击"提交"
- 操作结果：弹窗关闭，列表刷新，提示"提交成功"
- 异常处理：校验失败时标红错误字段；数量超过可用数量时提示"入库数量超过可用数量"

✅ 自然叙述（推荐）：
#### 发起入库申请：
- 点击"新增入库"打开申请弹窗，通过物资选择器从资产库选择标准物资，填写入库数量和税后单价后提交。
- 若任一明细税后价值超过 2000 元，单据进入财务审核；否则直接进入服务区确认环节。
- 提交成功后弹窗关闭，列表刷新。
- 数量超过可用数量时提示"入库数量超过可用数量"，未填写必填项时标红对应字段。
```

### 写作要求

- **具体数值写死**：所有数值直接写具体数字（"每页 20 条""倒计时 60 秒""保留前 3 后 4"），禁止"按配置""N 秒""具体数值见XX"等占位
- **UI 文案直接写**：提示文案、按钮文案、空状态文案直接出现在描述中（"暂无入库记录""正在导出..."），不单独列文案表
- **静态内容加引号，动态内容写来源**：静态标签、按钮、提示、空状态文案直接用双引号标明；动态展示需说明数据来自系统计算、接口返回、关联带出或用户输入
- **超长文本必须交代处理方式**：存在长名称、长备注、长说明、长渠道信息等场景时，写清截断、换行、滚动、悬停查看全文中的一种或多种处理方式
- **表格只用于映射关系**：数据字典、权限矩阵、状态映射、枚举定义用表格；区域说明、操作说明用自然语言或列表
- **标题下内容连续完整**：不补空栏目，不写只有标题没有实质内容的占位段
- **少用视觉强调**：不使用花哨强调，避免过多加粗；层级主要通过标题、编号、段落和列表表达
- **少讲原因，多写规则**：少写"方便用户""避免误解""用于承接"这类解释性表述，优先写展示字段、可点条件、状态变化、失败提示、边界限制
- **短句优先**：一句尽量只承载一层规则；能直接写成规格，不写成背景说明
- **区域感保留在页面开头**：可先用 1 段交代页面由哪些区域组成、各区域承担什么职责；后文仍按用户动作展开，不回到逐区盘点
- **不用伪排版**：正文采用完整段落，不用伪缩进字符，不手敲空格模拟首行缩进，不通过空行制造视觉层级

### 场景覆盖自检

每个动作写完后，检查是否覆盖以下场景（不存在的跳过）：

- **数据展示**：静态内容是否用双引号明确；动态内容是否说明数据来源；超长文本如何处理；无数据时显示什么
- **按钮与操作**：什么情况下可点击/不可点击；点击后的反馈（如加载动画、防抖、置灰）；操作成功后的变化；操作失败时如何提示
- **表单与输入**：输入限制（字符类型/长度/格式）；校验时机（实时/失焦/提交时）；错误提示文案
- **列表与加载**：加载方式（分页/无限滚动）；加载中的提示；无数据/加载失败/加载完毕状态
- **弹窗**：如何触发/关闭；点击遮罩层是否关闭；多个弹窗同时存在时的优先级和父子关系
- **异常与降级**：网络异常时如何处理；权限不足时如何提示；关键功能失效时如何降级
- **边界**：数值边界值、列表数据量上限、状态临界值、并发冲突

未覆盖的必须补充。

## 表达优化

- 混用三种表达：自然段（承接说明）、`1. 2. 3.`（顺序流程）、`·`（并列项）
- 有明确前后顺序、状态推进、处理链路时，优先用 `1. 2. 3.`
- 同层并列的展示项、校验项、限制项、异常项，优先用 `·`
- 自然段只承担语境交代和规则承接，不用大段自然段替代规格本身
- 通用 B 端规则（默认分页、常规空状态）无特殊要求不展开
- 不补写空洞栏目，不追求篇幅对称
- 标题文本中的中文编号前缀仅用于增强阅读结构感，不替代 Markdown 层级本身
- 禁用模板腔（"用于承载""用于支撑""需支持""按规范处理""同常规"）和模糊表述（"详见原型""待定""按业务规则"）
- 避免大白话或讲解腔（如"方便用户理解""避免用户误判""符合操作习惯"），正文保持专业、自然、可落地
- 除标题外少用加粗

## 长文档分块生成协议

PRD 预估超过 1000 行时启用。生成顺序：数据字典 → 业务流程 → 权限矩阵 → 页面说明（逐页）→ 验收标准 → 概述+范围。

每个页面：回读上游锚点 → 回读已生成页面摘要 → 生成（按动作组织）→ 保存 → 生成摘要 → 从上下文移除完整内容。

## 图表规则

- 复杂流程、状态流转、多角色交互、分支审批必须补图
- 风格与门禁遵循 `diagram-style.md`

## 防幻觉规则

- 每个功能点、字段、页面、区域、状态必须在 Feature List、Page Structure 或数据字典中有依据
- 找不到依据的标记 `[待确认]`，不得编造
- 允许扩展"怎么做"，不允许扩展"做不做"
