# Fix Processing Rules

本文件只定义 `fix` 阶段的判定与处理规则。

## 输入优先级

1. 用户显式给出的当前变更产物路径
2. `docs/project-status.json` 中与该产物对应的稳定基线
3. 项目目录扫描得到的候选产物

## 必做判定

1. 当前变更产物与稳定基线的关系
2. 修改层级判定
3. 回写判定

## 修改层级

- 仅文档表达修订
- 业务流程 / 结构层修订
- 页面承载层修订
- 能力边界层修订

## 回写判定

- 仅表达层：停留在当前产物
- 影响业务流程、判断节点、状态分支或系统结构：回写 `mind`
- 影响页面承载：回写 `page`
- 影响能力边界：回写 `feat`

## 影响分析

当确认需要回写时，必须先完成影响分析：

1. 变更提取
2. 传播路径判定
3. 影响范围枚举
4. 影响程度分级（局部 / 结构性）
5. 影响分析报告

## 回写粒度判定

- `局部`：只改对应章节或模块，不动其他部分
- `结构性`：必须全量回写该产物，不允许只改片段
- 若改动涉及模块间关系、排序、分组或全局性规则，直接视为 `结构性`
- 部分回写后，仍需补跑对应产物 skill 的自审部分

## 回写冲突处理

1. 锚点未确认：允许覆盖并更新锚点
2. 锚点已确认 + 来源明确为评审驱动：允许覆盖并更新锚点，并记录“评审驱动变更”
3. 锚点已确认 + 来源明确为用户手动修改：标记为冲突，不自动覆盖
4. 锚点已确认 + 来源不明：标记“来源待确认”，不自动假设为用户手动修改
5. 连锁冲突：逐层标记冲突，不自动级联覆盖

## 修正记录

- 如本轮已自动改动其他生成物，必须输出修正记录，写入 `docs/fix-records/`
- 结构参考优先读取：`shitpm/templates/fix-record.md`
- 若宿主不支持 bundle 路径，再回退到 `templates/fix-record.md`

## 模板使用与成稿清洗

- `fix-record.md` 只是修正记录骨架，不是可直接交付的半成品
- 模板中的占位字段、示例值、空表格行和教学说明，只能作为填写脚手架，不能原样进入最终修正记录
- 输出前必须删除未替换占位文本、空章节、空列表项和示例说明
