# Error Handling

## 通用规则

- 锚点冲突时停止推进
- 版本关系不清时停止推进
- 上游缺失时回退到对应阶段

## scope

- 若范围判定无法收敛，停止推进并输出待确认项
- 若子项目拆分结果无法形成清晰边界，停止推进，不进入任何子项目主链

## sum

- 若范围澄清结果不足以支撑分析，回退到 `scope`
- 若来源材料不足以整理关键流程、关键规则或角色边界，停止推进并要求补充材料
- 若分析过程中发现核心问题定义冲突，明确记录风险并停止继续下游

## mind

- 若思维导图中存在足以阻塞 `Feature List` 形成的重大待确认节点，先输出二次访谈问题清单，不直接进入 `feat`
- 若待确认节点在当前会话内已澄清，记录澄清结果再继续

## feat

- 若功能点混入步骤级内容，要求重写

## page

- 若页面结构无法从当前 Feature List 映射，要求先修 Feature List
- 若发现下游产物与页面结构冲突，不在本阶段直接修下游，交由 `rev` / `fix`

## prd

- 若 PRD 引用旧版上游，先修正 `stable_baselines`
- 若页面承载信息与 Page Structure 冲突，先回写或修 Page Structure
- 若上下文规模过大，先生成分块计划，不直接整篇输出

## rev

- 若显式给出的当前文件与稳定锚点不一致，先做版本判定
- 若当前评审对象未判清，不得默认按 PRD 继续
- 若待处理问题数量过多，不得一次性全部抛给用户，先做归并和阻塞度排序
- 若评审团意见与稳定锚点、正式规则或合同冲突，以稳定锚点和正式规则为准
- 若评审问答已改变事实源但未正式路由到 `fix`，不得宣称问题已闭环

## fix

- 若当前变更产物路径未显式给出且存在多版本，停止推进
- 若当前变更产物与稳定锚点关系不清，先修状态文件再继续

## mock

- 若稳定 PRD 不足以支撑页面表达，回退到 `prd`
- 若页面承载事实不足以支撑原型结构，回退到 `page`
- 表现层问题可停留在本阶段修订，不得擅自新增 `Page Structure` 中不存在的主页面或关键区域

## note

- 若未判定初始化标注或增量更新模式，停止推进，不得擅自修改原型 HTML
- 若发现需求冲突，回退到 `prd` 或 `mock`
- 若只是标注内容问题，停留在本阶段修订
