# Error Handling

## 通用规则

- 锚点冲突时停止推进
- 版本关系不清时停止推进
- 上游缺失时回退到对应阶段

## pm-scope

- 若范围判定无法收敛（需求边界持续扩大或相互矛盾），停止推进，输出待确认项，等待用户确认后再继续
- 若子项目拆分结果无法形成清晰边界，停止推进，不进入任何子项目主链

## pm-analysis

- 若范围澄清结果不足以支撑分析，回退到 `pm-scope`
- 若来源材料不足以整理关键流程、关键规则或角色边界，停止推进并要求补充材料
- 若分析过程中发现核心问题定义冲突，明确记录风险并停止继续下游

## pm-mm

- 若思维导图中存在足以阻塞 `Feature List` 形成的重大待确认节点，先输出二次访谈问题清单，不直接进入 `pm-fl`
- 若待确认节点在当前会话内已澄清，记录澄清结果再继续

## pm-fl

- 若功能点混入步骤级内容，要求重写

## pm-ps

- 若页面结构无法从当前 Feature List 映射，要求先修 Feature List

## pm-prd

- 若 PRD 引用旧版上游，先修正 `stable_baselines`
- 若页面承载信息与 Page Structure 冲突，先回写或修 Page Structure
- 若上下文规模过大，先生成分块计划，不直接整篇输出

## pm-rv

- 若显式给出的当前文件与稳定锚点不一致，先做版本判定

## pm-fix

- 若当前 PRD 路径未显式给出且存在多版本，停止推进
- 若新 PRD 与稳定锚点关系不清，先修正状态文件再继续

## pm-pt

- 若稳定 PRD 不足以支撑页面表达，回退到 `pm-prd`
- 若页面承载事实不足以支撑原型结构，回退到 `pm-ps`
- 表现层问题可停留在本阶段修订，不得擅自新增 `Page Structure` 中不存在的主页面或关键区域

## pm-pa

- 若未判定初始化标注或增量更新模式，停止推进，不得擅自修改原型 HTML
- 若发现需求冲突，回退到 `pm-prd` 或 `pm-pt`
- 若只是标注内容问题，停留在本阶段修订
