# PRD Review Checklist
本文件只定义 `prd` 阶段生成完成后的自审与门禁检查。生成完成后读取；不要把写作规则混在这里。

## 一致性校验

全部页面生成完毕后执行：

- 功能覆盖：Feature List 每条"本期"功能点是否有对应描述
- 页面覆盖：Page Structure 每个页面是否有对应章节
- 字段覆盖：Page Structure 各区域 `关联字段` 是否在数据字典或页面章节中体现
- 字段追溯：上游字段线索是否在数据字典中有去向
- 字段一致性：同一字段在不同页面的展示格式和校验规则是否一致
- 权限一致性：权限矩阵与各页面按钮可见性是否匹配
- 状态一致性：状态流转图与各页面状态筛选/展示是否一致
- 交叉引用：FL-__/PS-__/DD-__ 是否指向真实条目
- 模糊表述：全文搜索"需支持""需考虑""按规范处理""详见原型""同常规""待定""按业务规则""具体数值见配置"，确保零命中

## 写作质量检查

- Markdown 标题层级是否保持稳定；标题文本中的中文编号前缀是否仅用于增强阅读结构感，且未跳级
- 页面内部按用户动作组织（动词短语），非 UI 区域盘点
- 每个动作都覆盖触发、过程、结果、异常这四类信息，但不要求写成固定四段格式
- 简单动作一段话带过，复杂动作按复杂度展开
- 页面开头可用一小段交代真实区域职责，但后文不得退回按区域逐项说明
- 有顺序的流程是否使用 `1. 2. 3.`，并列规则是否使用 `·`，节奏是否自然
- 正文是否保持规格说明口径，避免大白话、解释腔、设计说明腔
- 标题下内容是否连续、完整，无空栏目、无占位段
- 是否避免过多加粗或花哨强调
- 是否未使用伪缩进字符、手敲空格模拟首行缩进、空行制造视觉层级
- 所有数值写死具体数字，无占位符
- UI 文案直接写在描述中，无单独文案表
- 静态内容是否用双引号明确；动态内容是否说明数据来源
- 超长文本场景是否写清截断 / 换行 / 滚动 / 悬停查看全文的处理方式
- 按钮是否写清可点击/不可点击条件、点击反馈、成功变化、失败提示
- 表单是否写清输入限制、校验时机（实时 / 失焦 / 提交时）和错误提示文案
- 列表是否写清加载方式、加载中提示、无数据/失败/完毕状态
- 弹窗是否写清触发/关闭方式、遮罩层行为、多弹窗优先级
- 异常是否写清网络异常、权限不足、关键功能失效时的降级处理
- 表格只用于数据字典、权限矩阵、状态映射、枚举
- 无模板腔、无模糊表述、无空洞栏目
- 按 `prd-generation-rules.md` 中的场景覆盖自检清单逐动作检查

## 门禁规则

- 一致性校验为必做门禁，不是建议
- 功能/页面/字段/交叉引用/模糊表述/场景有缺口，不得进入状态更新
- 仅表达不足：留在当前稿件修正
- 问题来自上游（Feature List / Page Structure）：停止推进，回退到 `feat` 或 `page`

## 角色视角自审

生成完毕后从三个视角审读：开发（能否直接写代码）、测试（能否直接写用例）、设计（能否直接画原型）。

上游事实源缺失不得脑补，标记 `[待确认]` 或回退上游。
