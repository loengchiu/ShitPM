# Review Rules

本文件定义 `rev` 阶段的评审规则。

## 模式判定

- 自检模式：用户未提及具体角色，AI 独立完成一致性检查
- 多角色评审模式：用户提及角色、评审会或开发对齐，切换到多角色视角

## 自检模式

### 输入优先级

1. 用户显式给出的当前文件路径
2. `stable_baselines`
3. 目录扫描结果

### 评审对象判定

- 用户明确指定对象时，按指定对象评审
- 用户只说“开评审”时，按当前阶段或最近产物判定
- 无法判清时，先要求确认对象，不默认按 PRD 处理

### 必做检查

1. 版本关系
2. 产物一致性
3. 分层评审结论（质量层推进建议）
4. 回写判定

### PRD 补充检查

当评审对象包含 PRD 时，额外检查：
- 功能覆盖
- 页面覆盖
- 字段一致性
- 权限一致性
- 状态一致性
- 交叉引用完整性
- 场景覆盖
- 模糊表述
- 边界场景
- 字段线索传递完整性
- 动作组织是否符合当前 PRD 规则（按用户动作，不按 UI 区域盘点）
- 单个动作是否写清触发条件、操作过程、操作结果和异常处理
- 页面开头若保留区域描述，是否仅用于帮助理解页面职责，而未退回逐区说明
- 是否按内容性质混用自然段、`1. 2. 3.`、`·`，而非全文单一句式
- 正文是否保持规格说明口径，避免大白话、解释腔、设计说明腔
- 具体数值与 UI 文案是否已直接落到正文，而不是用占位或外链代替

## 多角色评审模式

### 角色与默认关注域

- 业务经理：业务目标、版本价值、范围取舍
- 项目经理：排期依赖、风险、跨团队协作
- 产品经理：闭环、边界、异常与规则
- 前端：交互复杂度、状态、组件边界
- 后端：实体、接口、权限、状态机
- UI：信息层级、视觉承载、实现一致性
- 测试：可测试性、边界、异常、回归风险

### 评审团组合模板

- 需求边界评审团
- 页面承载评审团
- 可开发性评审团
- 上线风险评审团

### 结论判定

- `通过`
- `带条件通过`
- `阻塞`

## 评审后问答模式

- 不一次性把所有问题丢给用户
- 逐条收敛，默认每轮只推进最关键的 1 个问题；强相关时最多同轮推进 3 个
- 问题较多时先归并、去重并按阻塞度排序

### 问题定性

- 真实风险
- 过度担忧
- 待观察项

无法判定时，默认先按真实风险处理。

## 与 fix 的联动

如果评审问答后的答案已经改变事实源，必须继续路由到 `fix`。

## 冲突优先级

- 若评审团意见与稳定锚点、正式 skill 规则或对应合同冲突，以稳定锚点和正式规则为准
- 评审团可以指出风险、建议回写，但不能直接覆盖事实源
